# 我希望用一篇文章帮你叩响AI辅助编程的大门

不积跬步无以至千里，欢迎来到 AI 时代的编码实战课

一、场景引入：‍2 小时，万行代码，然后呢？

几天前做了一场几百号人的 AI 辅助编程培训，大概是 2 天教辅助编程，半天教 LangChain 基础。到第三天下午，学员 2 小时就搭建了一套知识问答 RAG 前后端 —— 万行代码，包含了长会话、会话历史管理、反馈评价、反思升级、监控、多模态上传。多少有点令人惊讶。

但学员们对上下文的认知，以及如何提高生成质量的认知，其实也无法仅用两三天的培训补齐。而且很可能短期内也没诉求需要去补齐。但这个很重要，所以想着写篇文章。如果一时看不懂，多实践多观察多学习，总会有需要和会看懂的时候。

这篇文章有两条主线：

1. 从培训实践出发 — 讲清楚为什么我们需要从纯提示词一路演进到分层 plan + 子代理委派的重量级框架

2. 从自建框架出发 — 讲清楚搭建一套 SDD 框架时，Agent、Skill、Script 这三类组件的设计实践

二、纯提示词时代：‍模型已经掌握了你的工程手段

一开始我们使用纯 Vibe Coding 和 RIPER-5。通过实践可以感知到，现在的 code 平台纯 Vibe Coding 已经很强 —— 它懂得去扫代码，结合上下文开始做计划，分 task 后开始执行。这样上下文也少，生成质量其实不错。如果说 RIPER-5 是 80 分，那纯 Vibe Coding 已经能到 75 分了。

通过实践我们还可以感知到，我们想提高生成质量、保障模型不偷懒，需要使用一些工程手段。而我们在 25 年所采用的一些有效的工程手段，已经被模型所掌握。所以在使用最好的模型的前提下，纯提示词的约束效果和这些工程手段的差距已经不大了。

但纯靠模型/提示词去完成工作，依旧有很多不足。比如缺少 plan with file —— 当我们发现上下文太大时，没有上个阶段的产物可以依赖，只能顶着压力继续用一个会话沟通，防止丢失上下文。同时，聊天过程中产出的方案也缺少全貌，不是很好阅读。此外，虽然模型可以完成所有工作，但一些明确的规则 —— 比如检查当前任务的执行进度 —— 其实通过固定的 Script 就能实现。交给模型做，一方面有 token 成本，另一方面也会增加不必要的上下文噪音。所以也需要把可以识别的固定规则沉淀成工具，通过调用完成，降低模型工作的 scope，保证上下文方向的准确。

三、轻量级 SDD 框架：‍千行级别的舒适区

所以我们需要引入轻量级的 SDD 框架，比如 Spec-Kit 和 OpenSpec，这两个框架其实就足够满足日常迭代的需求开发了。简单来说，基本就只有 brainstorming(澄清需求) → plan(制定实现步骤) → task(拆分实现步骤) → impl(实现代码)。

brainstorming 存在的意义其实是扫个底。大部分 case 我们用默认的选项就可以，看似在选方案，但感觉核心更偏开卡流程 —— coding 平台在向用户阐述它对需求的理解。plan 阶段则是结合项目和用户的需求输入，把完整的实现方案写下来。task 则是拆分和逐步生成，降低单次生成的复杂度，支持断点续传。但更重要的是，我觉得它在强调 AI 要做哪些事、别漏了 —— 拆了 task 就有明确的执行（检查）事项。impl 就不用说了，逐个实现代码。

而这些框架最新的升级，也就是围绕项目级 spec 约束（架构可执行/优秀实践沉淀），以及执行后的一致性校验（重入的判断，用再次检查为潜在的生成走偏兜底），主链路没太大变化。

通常这样的实现是好的，因为需求不大时，总的上下文是有限的，AI 是能从繁杂的上下文中推断出当下要做的事情是什么。量不大为什么上下文会繁杂？因为整个会话框都是 AI 对话的上下文 —— brainstorming 时的交谈过程，项目的 SPEC 和代码上下文，各阶段输出的内容，以及实现代码时的生成过程。可能量不大，但全是。如果 AI 不具备理解上下文的推理能力，比如仅看各阶段的 token 数判断哪些内容重要，实现过程的重心就可能偏离，导致忘记最一开始的约束。例如，有约束只改 domain 的代码，因为就一点点 token 占比，到了最后可能就在实现便捷和规范之间选择了便捷，去改了别的目录的内容。而这，是非常危险的。

好在上了 C4.6 Opus 后这种推理能力是具备的。所以当下单个会话框 + OpenSpec 这种框架生成一个 5-6k 行代码的前后端项目，7-8 成能一把过，剩下的也就十几分钟把问题反馈给它就能修好。这样的生成质量下，SDD 轻量框架的测试检测就有点脱裤子放屁了 —— 一次生成能准确何需测试来兜底（百行的代码生成真的大概率不需要测试了）。千行级别可能还是需要 lint 检查和测试覆盖，毕竟也不是达到 20W 上下文才会失去中心，上下文有个 7-8W token 往上就越来越容易失去重心。就像人一样，一下子塞进来的信息越多，你越没法快速定位重心是什么，可能就比较看重大篇幅和高频出现的内容。

专业点说可以叫 "注意力稀释效应"，说的是上下文越长，LLM 对每个信息片段的关注度越低。一个 5000 token 的上下文中的关键指令，比一个 50000 token 上下文中的同样指令，会被更准确地执行。有时候我们也叫 "Lost in the middle" —— 大模型在长上下文中容易漏掉中间的内容。

但是，如果我的需求要生成的代码量级是万行级别的（所谓的长上下文），我们来算一笔账：

总上下文 ≈ 固定开销 + 累积开销

固定开销（会话全程存在）：
  - System prompt + CLAUDE.md + Rules    ≈ 2,000-5,000 token
  - Plan / Spec 文档                     ≈ 3,000-10,000 token
  - Brainstorming 对话历史               ≈ 5,000-15,000 token

累积开销（随生成线性增长）：
  - 每行代码输出                         ≈ 10-15 token
  - 每行代码对应的工具调用开销            ≈ 5-8 token
    （Read/Edit/Write 的调用结构、文件路径、返回结果）
  - 模型的推理/规划文本                   ≈ 3-5 token/行

单行综合成本 ≈ 18-28 token（取中值 ~23 token/行）

但这还没算一个关键因素 —— 上下文的滚雪球效应：‍模型在生成第 5000 行时，前 5000 行的代码和工具调用结果都还在上下文里。所以实际是：

生成第 N 行时的上下文大小 ≈ 固定开销 + N × 23

生成到第 5,000 行时 ≈ 2W + 11.5W ≈ 13.5W token（已接近 20W 限制）
生成到第 8,000 行时 ≈ 2W + 18.4W ≈ 20.4W token（触顶）

万行代码？不做任何压缩的情况下，数学上就过不去。而当下大部分 code 平台的会话窗口虽然有 100W token，但背后的模型上下文限制是 20W token，超出的部分必须靠压缩来腾挪空间。

不过，培训时确实大概有不到十分之一的学员，一把生成了没 bug 的万行代码项目。因为即便我反复强调要用 agent/会话框隔离任务，还是大量的学员在一个会话框一撸到底。万行代码级别的生成，会话框上下文一定是超了的，更别说他们多数还是多轮迭代。但这就不得不说 Cursor 在上下文优化这块确实下了很大功夫。当前 code 平台的上下文压缩主要靠这几种手段组合使用：

1. 对话历史摘要（Conversation Summarization） — 当上下文接近阈值时，把早期的对话轮次压缩成一段摘要。比如你和 AI 讨论了 20 轮需求澄清，压缩后可能变成一段 500 token 的"需求共识摘要"。信息密度提高了，但细节不可避免地丢失。

2. 工具调用结果裁剪（Tool Output Pruning） — 模型在执行过程中会产生大量工具调用结果（Read 文件返回的内容、Grep 搜索结果、Bash 执行输出等），这些结果在使用完后其实不需要完整保留。平台会把已消费的工具结果截断或摘要化，只保留关键信息。

3. 生成过程日志清理（Generation Log Cleanup） — 模型在生成代码时会产生大量中间推理文本（"我先看看这个文件..."、"这里需要注意..."），这些在代码生成完成后价值很低。平台会在合适的时机清理这些中间日志，只保留最终的代码产出。

4. 滑动窗口 + 选择性保留（Sliding Window with Pinning） — 保留最近 N 轮完整对话，更早的对话只保留摘要。但某些关键信息（如用户的初始约束、plan 文档路径）会被"钉住"不压缩。

5. 智能上下文选择（Smart Context Selection） — 不是把整个项目都塞进上下文，而是根据当前任务动态选择相关文件。比如你在改 OrderService.java，平台会自动扫描并加载 Order.java、OrderRepository.java 等相关文件，而不是加载整个 src/ 目录。

这些手段的本质是把"物理上下文窗口"（100W token 的会话窗口）和"有效上下文密度"解耦 —— 窗口很大，但真正送给模型的 20W token 里，信息密度尽可能高。这也是绝大部分同学抱着 Cursor 基本也能躺平的基础。

不过前边的例子是因为学员生成的方向是比较明确的需求，期间没有干别的，所以上下文的删减和压缩还是比较有效的。但如果在一个会话框里，一会儿聊需求，一会儿查 bug，一会儿生成查询 SQL，做着各种各样不同的事情，那么这种上下文的删减和压缩效果就会大打折扣。

总的来说，上下文压缩确实能很有效地提高生成质量，而且压缩越早发生越好，并不是一定要到 100% 才压缩，既省钱又提高质量。但有子代理委派依旧可以用上下文压缩，并且理解完它的运作机制，其实很直观就能感知到它的效果会比单纯的上下文压缩来得更好。面向真正的长上下文生成，大概率还是不能只依靠上下文压缩的。

四、重量级框架登场：‍子代理委派与企业级挑战

这些轻量级框架很难做一个相对完备的长上下文应用。这时候重量级的框架就登场了。比如 superpowers（框架本身是轻的，它类似微内核需要插上各种 skills，但它的流程其实是重的）。核心的步骤：‍brainstorming(澄清需求) → plan(制定实现步骤) → task(拆分实现步骤) → impl(实现代码)(TDD 循环增强) → 一致性检测 & 自动化测试修复（可选）。

子代理委派：‍万行级别的突破口

在 task 环节开始使用 sub-agent 子代理委派/集中式编排的模式。简单说，就是由执行了 brainstorming(澄清需求) & plan(制定实现步骤) 的会话做主 agent，然后通过后台起并调度子代理会话，开始做具体的编码实现任务。

这样做有两个好处。首先，通过会话隔绝了每个 task 实现时的过程内容，变相提高了主代理的上下文大小 —— 执行过程的内容分出去后，能生成的代码行数就增加了。同时，单个 subagent 因为也没有主代理前面 brainstorming(澄清需求) 和 MCP 啥的内容，所以也相对干净。干净上下文就少，少生成就准确。

但是信息不管是分割还是压缩，都会带来缺失。拆完 task 然后把 task 交给 subagent 时，subagent 是否具有实现该功能完整的上下文？subagent 并行工作时是否会有冲突？（虽然 superpowers 说不要并行，但 4.6 Opus 有时会控不住 —— Opus 太聪明了，会把可以并行的 task 并行处理，正确但不可靠，不听话就很难信任。）这些是客观存在的风险，核心还是我们创建子 subagent 时，不见得给的就是一个完备上下文的内容。

虽然 TDD 的设计也可以提高代码的完成度，但我觉得有效的部分可能不是 TDD 的方法论，而是出现了多轮重试。就像我们在发现问题时重新调用模型一样 —— 你重新调就强调了某块功能，这时候模型大概率就能生成更好。TDD 的重构机制没有很好的复杂设计能力支持，很容易导致补丁代码的出现，进而越走越偏。

最后就需要有一些兜底操作，插拔一些比如 Playwright/Agent-Brown 这些技能去做测试，进一步自动查缺补漏。但做完了其实还是会有一些遗漏，思来想去多半还是上下文在作怪 —— 模型会选择性遗忘，或者偷懒。

但总的来说，有 sub-agent 子代理委派/集中式编排的模式在，确实提高了生成代码行数的上限，几万行一把过也不是梦。我个人的实践，最多是实现一个小工具，一把 2W 多行没 bug，当时用的还是 4.5 Opus，真的有种 aha 时刻的感觉。

企业级的挑战：‍集中 plan 的瓶颈

但是，当这个诉求放到了企业级，superpowers 就有点力不从心了。

首先，OpenSpec 这种 k 行级别的框架，开发者还是需要协作的 —— 追加上下文 & 方案检验。所以很多企业级的信息就由人裁剪给了 AI。比如当前功能需要依赖的 user 信息从 user 二方服务的 userService 上获取，以及 RPC 框架用自研的 XXXRPC，其作为 CLI 的使用方式如下（可能是让其参考某段已有代码）。把应用间的依赖关系、哪个应用有什么能力、以及自研框架的使用方法都塞给了 AI。有点一等的人才在搞模型和算法，中间全是 AI，到了最后一公里由人来给 AI 打下手（这是个玩笑，因为底下如果还需要人最后一公里，其实 AI 的生产力是会严重受限的）。

但是万行级别的不行。万行级别人其实已经关注不了方案粒度，所以由人来补充上下文可能不全。但不靠人拆解，企业级的自研框架可能百八十个，一大堆配置信息，企业的应用架构与领域划分也是几十上百领域、成百上千的服务。这些内容如果一股脑直接给 code 平台，那会话直接炸了。即便我们通过一定匹配规则仅捞出一个几万行代码规模需求需要考虑的，也可能有 4-5 个框架 & 服务的信息需要给 AI。

可能有些同学会觉得：‍如果代码里面已经有了，不是会自己扫使用方式和已经具备的领域服务接口信息给补充到上下文吗？事实上确实也会。如果使用方式刚好匹配，确实就会扫过来，这样信息会少一些。但它还是会加载到上下文中，并且 AI 不能区分当前需求其实有更好的配置方式，只会抄已有的 —— 因为这些信息在模型的训练中并不存在，无论什么形式必须在上下文中。

而问题的关键就在这里：‍task 确实可以丢给 sub-agent 来执行，但 plan 阶段是规划当前需求整个项目的代码实现，所以它需要上边提到的所有上下文 —— 用什么技能、怎么用，依赖哪些服务、什么接口，怎么解析，提供接口给了哪些服务，当前变更对调用方有什么影响，以及当前需求实现相关的内容。所以 plan 阶段很大，大到把它单独放在一个 agent 的会话都完成不了 —— 因为我们是集中 plan 的。

从集中 plan 到分层 plan

但是我们的生成是整个项目生成的，plan 的信息就一个都少不了。所以就还要进一步拆分，比如把单个服务拆成 DDD 的 OHS → Application → Domain → Infr。这样在 plan Domain 时，就只需要考虑企业应用架构里需要依赖的领域和本身的功能需求。Infr 只需要关注自研框架 & Domain 需要它实现的逻辑。Application 只需要关心 Domain 提供的能力和当前需求。OHS 只需要关心 Domain & Application 提供的能力，以及上游依赖服务对接口的诉求。

这样分成 4 个 plan 后，相当于原本 plan 的上下文容量增加了 3-4 倍。而且这种增加比单个会话硬撑的效果更稳定 —— 因为单次的上下文还是少的，上下文越少模型越聪明。是不是就能解决大部分场景了？而只要 plan 能过，task 的分发就不是问题 —— 模型现在自己已经知道这个流程，并且执行得很好。

五、思路有了，怎么做：‍Agent、Skill、Script 三件套

前面四章讲的是"为什么" —— 为什么需要从纯提示词一路演进到分层 plan + 子代理委派。这一章讲"怎么做" —— 当你决定自建一套 SDD 框架时，Agent、Skill、Script 这三类组件分别承担什么角色，怎么设计才能让整个系统跑起来。

以后端为例，首先我要拆分 plan，借助 subagent 实现三个目标：隔离 —— 子 plan 对主 agent 上下文的污染；约束 —— 在子 plan 加载相关的约束，比如 Infr 加载相关自研框架的使用手册和实现规范，保障按要求实现代码；复用 —— 在多个项目多个需求能复用这个 subagent 去做专项的工作。

但各层之间是有依赖的，所以需要约束每一层 plan 的产出物，保证下游消费者能完备获取上游产出的内容。同时有依赖就还要设计一套状态检查机制 —— 不能依赖的文件有了，也不看生成的是否符合预期就一直往下走。所以需要有一套工作流的编排和各节点的状态管理，以及产出物上要能有明显标识能判断是否已经 ready。此外，我还需要观测执行过程，所以至少需要对每个文件有个 ID 标识，以便于在 code 平台加载完文件后，上下文中有这个文件 ID，我可以通过 ID 知道哪些文件在什么时候被加载了。

以下以我们实际搭建的 DDD 四层架构框架为例，拆解每个组件的设计实践。

5.1 Agent：‍独立上下文的执行单元

Agent 是子代理委派的载体。每个 Agent 有自己独立的上下文窗口、独立的工具权限、独立的系统提示词。在我们的框架中，Agent 分为两类：

Thinker Agent（设计者） —— 只做设计，不写代码。产出是设计文档。

Worker Agent（执行者） —— 只按设计写代码，不修改设计。产出是代码文件。

这个分离不是为了好看，而是为了解决一个实际问题：‍如果让一个 agent 既做设计又写代码，它的上下文会被设计过程中的推理、反思、修改历史占满，等到真正写代码时，上下文已经很拥挤了。分离后，Worker 拿到的是一份干净的、经过验证的设计文档，上下文利用率最高。

为什么 Thinker 要禁止 Edit？因为 Thinker 的职责是创建设计文档，不应该去修改项目中已有的文件。如果 Thinker 能 Edit，它可能会"顺手"去改已有代码，这就越权了。通过 disallowedTools:‍ Edit 在 agent 定义层面硬性约束，比在 prompt 里写"请不要修改已有文件"要可靠得多 —— prompt 约束是软性的，模型可能会"合理化"地绕过；工具权限约束是硬性的，模型想绕也绕不过。

本质上，Skill 和 Agent 的 frontmatter 头部本身就是一种"把固定逻辑沉淀成代码"的实践 —— 在文件上打标（permissionMode、disallowedTools、maxTurns、skills、disable-model-invocation 等），code 平台识别这些标记后做出硬性约束。这比在 prompt 里写一堆"请你不要..."然后祈祷模型听话，要稳固得多。

反思循环

每个 Thinker Agent 完成设计初稿后，必须进入 2-3 轮反思循环，每轮包含四个验证步骤：

1. 目标覆盖验证 — 回到 MISSION 中的每个工作项，逐条检查是否有对应的设计产出，并引用具体章节内容作为证据

2. 下游可消费性验证（或上游契约一致性验证）— 站在下游 agent 的角度审视：‍导出契约完备吗？签名足够具体吗？行为描述够详细吗？

3. 规范符合性验证 — 对照编码规范检查命名、模式、约束

4. 实现推演验证 — 切换视角为 Worker，在脑中按方案描述写出完整实现代码，如果写到某一行发现方案在特定数据状态下会导致运行时异常，说明方案本身有缺陷

反思循环的关键不在于"做了几轮"，而在于每轮必须产出具体的验证记录。"已检查，没问题"不算验证，必须引用方案中的具体内容作为证据。这是防止模型"走过场"的核心手段。

上下文完备性：‍委派时的信息传递

前面提到子代理委派的核心风险是上下文完备性 —— 我们创建子 subagent 时不见得给的就是一个完备上下文的内容。我们的应对方式是：在构建 subagent prompt 时，按 MISSION → TEMPLATE → CONTEXT → OUTPUT 四个区块组装完备的上下文。

MISSION（工作目标）
 — 不是把原始需求丢过去，而是主 agent 从 assessment.md 中提炼出结论先行的工作目标描述。2-4 句话说明这一层要做什么、为什么要做，再列出具体的可验证工作项。这些工作项会成为反思循环中逐条验证的基准。
TEMPLATE（产出骨架）
 — 指向设计文档模板的路径，让 agent 自行 Read 加载。模板定义了产出的结构，确保所有 agent 的输出格式一致。
CONTEXT（输入文档）
 — 这里有一个关键设计：上游导出契约必须完整内联到 prompt 中，而上游设计文档全文改为提供路径让 agent 自行按需 Read 加载。为什么？因为导出契约是下游层的硬依赖，必须确保 agent 一定能看到；而上游设计文档全文可能很长，全部内联会浪费上下文空间，按需加载更高效。
OUTPUT（产出位置）
 — 明确写入路径和 frontmatter 格式要求。

这四个区块的设计目标是：让子代理在没有主 agent 完整上下文的情况下，依然能做出正确的设计决策。MISSION 提供了"为什么做"，TEMPLATE 提供了"怎么输出"，CONTEXT 提供了"基于什么做"，OUTPUT 提供了"产出到哪里"。

5.2 Skill：‍流程编排与渐进加载

Skill 是 Claude Code 插件体系中的核心概念，可以理解为"可被调用的流程定义"。在我们的框架中，Skill 分为两种类型：

自主加载型 Skill（模型自动触发）

这类 Skill 的 frontmatter 中没有 disable-model-invocation:‍ true，意味着模型可以在合适的时候自动加载使用，也可以由用户或主 agent 显式调用。我们框架中的大部分 Skill 都属于这一类：

Skill

职责

触发方式

thoughtworks-skills-ddd
（Decision-Maker）

后端入口路由，编排澄清→评估→设计→编码

用户调用或 session-start hook 触发

thoughtworks-skills-ddd-clarify
项目上下文扫描 + 结构化需求澄清

thoughtworks-skills-ddd-thought
设计编排，按 DAG 拓扑序派发 Thinker subagent

被 Decision-Maker 调用

thoughtworks-skills-ddd-works
编码编排，按 DAG 拓扑序派发 Worker subagent

自主加载型 Skill 的好处是灵活 —— 模型可以根据上下文判断什么时候该调用哪个 Skill，不需要用户手动触发每一步。比如当用户说"帮我实现用户注册功能"时，session-start hook 注入的触发索引会引导模型自动调用 Decision-Maker，Decision-Maker 再依次调用 clarify → thought → works。

任务型 Skill（替代 Commands）

这类 Skill 的 frontmatter 中标记了 disable-model-invocation:‍ true，意味着模型不会自动触发它，只能通过显式调用（如 /thoughtworks-skills-java-spec domain）来使用。在我们的框架中，thoughtworks-skills-java-spec 就是这种类型：

name: thoughtworks-skills-java-spec
description: Use when working on Java Spring Boot DDD projects...
argument-hint: "<layer> e.g. domain, application, ohs, infr..."
disable-model-invocation: true

它的作用是一个规范路由器：‍接收层级关键词（如 domain、ohs），路由到对应的规范文件（references/common.md + references/domain.md）。为什么要设计成任务型而不是自主加载型？因为编码规范是按需加载的 —— 只有在 agent 真正需要某一层的规范时才加载，避免一次性把所有层的规范都塞进上下文。而大模型自主可能在扫到时就加载了，比如代码侦探阅读代码仓时就加载进来了（虽然 Claude Code 也有 rule 可以通过 path 做分层的 spec 加载，但实际使用下就会发现，这个加载不够灵活 —— 该加载的 case 没加载，不该加载时加载。毕竟不是你读到了就该加载。）

这个 Skill 被配置在每个 Agent 的 skills 字段中（如 skills:‍ [thoughtworks-skills-java-spec]），当 Agent 启动时，Skill 的路由规则会自动注入到 agent 的上下文中。Agent 的系统提示词中会指导它：‍按照该技能的路由规则，使用 domain 关键词匹配，通过路由表中的 markdown 链接用 Read 工具加载对应的规范文件。这样，规范的加载是渐进式的 —— agent 先看到路由规则，再按需加载具体规范，而不是一开始就把所有规范都塞进来。

渐进加载的设计哲学

渐进加载不仅体现在规范文件上，整个框架的信息加载都遵循这个原则：

每一层只加载自己需要的信息，不多不少。session-start hook 只注入一个几十 token 的触发索引（"当用户需求涉及 DDD 时，调用哪个 Skill"），而不是把整个框架的所有规范都塞进去。Decision-Maker 只加载编排流程，不加载具体的设计模板。Thinker Agent 只加载自己那一层的规范和上游契约，不加载其他层的。这种层层递进的加载方式，确保了每个环节的上下文都是精准的、不冗余的。

5.3 契约驱动的跨层一致性

分层 plan 的一个核心挑战是：‍各层独立设计，怎么保证它们之间是一致的？目前的做法是契约驱动。

每个设计文档包含两个关键区块：

导出契约
 — "我提供什么"。Domain 层导出聚合根 API、值对象定义、仓储/事件发布/防腐层接口签名。Application 层导出 ApplicationService 方法和 Command 定义。
依赖契约
 — "我需要什么"。描述从哪里取（上游层的导出契约+项目已有能力分析）、怎么取（通过哪个接口/方法）、取来干什么（用于实现什么业务逻辑），并用目标来点亮对获取信息的解读方式 —— 同一个接口签名，在不同的业务目标下，消费方关注的行为细节是不同的。

依赖契约不是简单地"抄"上游导出契约的签名，而是要带着目标去引用。比如 Application 层依赖 Domain 层的 OrderRepository.save(Order)，它的依赖契约不能只写"调用 save 方法"，而要写清楚：‍从 Domain 导出契约的接口签名表中获取 save 方法签名，用于在"创建订单"用例中持久化新创建的 Order 聚合根，关注的行为是 insert-or-update 策略（首次创建时 insert，重试时 update）。这样即使上游修改了 save 的行为描述，下游也能根据自己的目标判断这个修改是否影响自己。

校验 Script：‍确定性的一致性保障

契约匹配不能靠模型"看一眼觉得对"，必须用确定性的 Script 来校验。我们的 ddd-output-validate.sh（585 行纯 bash + awk）执行以下校验规则：

结构完整性校验（S 系列）：

S1:‍ frontmatter 必填字段完整（layer、order、status、depends_on、description）
S2:‍ frontmatter 的 layer 值与文件名前缀一致
S3:‍ 结论章节存在且有非空内容
S4: 实现清单表格存在且数据行数 > 0
S5:‍ 导出契约章节存在（仅 domain 和 application）
S6:‍ 依赖契约章节存在（仅 infr、application、ohs）
S7:‍ 导出契约中每张子表至少有一行数据

内部一致性校验（I 系列）：

I1:‍ 导出契约中的方法签名能在正文中找到对应定义（防止契约表有但正文没有的"幽灵签名"）
I2:‍ 实现清单行数 ≥ 导出契约行数 + 依赖契约行数（确保每个契约项都有对应的实现项）

跨文件契约匹配（C 系列）：

C1:‍ Infr 依赖契约的 Repository 实现签名 ⊆ Domain 导出契约的接口签名
C2:‍ Application 依赖契约的聚合根 API 签名 ⊆ Domain 导出契约的聚合根与实体 API
C3:‍ Application 依赖契约的 Repository 接口签名 ⊆ Domain 导出契约的接口签名
C4:‍ OHS 依赖契约的 ApplicationService 方法签名 ⊆ Application 导出契约的 ApplicationService 方法
C5:‍ OHS 依赖契约的 Command 定义逐行全匹配 Application 导出契约的 Command 定义

这些校验规则的核心思想是：下游层的依赖契约必须是上游层导出契约的子集。如果下游依赖了上游没有导出的东西，说明设计有问题 —— 要么上游漏了，要么下游越权了。

5.4 Script：‍把确定性的事交给确定性的工具

在整个框架中，有三类操作是确定性的，不需要模型推理：

1. 状态管理 — 某一层是 pending / in_progress / done / failed，这是一个有限状态机

2. 契约校验 — 签名匹配是字符串比较，不需要语义理解

3. 进度查询 — 当前哪些层完成了、哪些还在进行中

这三类操作如果交给模型做，一方面有 token 成本，另一方面模型可能会"创造性地"解读状态（比如把一个 failed 的层"合理化"为"基本完成"）。用 Script 做，结果是确定的、可审计的。

ddd-workflow-status.sh（460 行）是状态管理的核心 Script，支持五种模式：

# 初始化状态文件（只注册需要开发的层）
bash ddd-workflow-status.sh <idea-dir> --init <idea-name> domain infr application ohs

# 设置某层状态
bash ddd-workflow-status.sh <idea-dir> --set domain in_progress

# 非阻塞检查上游是否 ready
bash ddd-workflow-status.sh <idea-dir> --check-upstream application

# 非阻塞检查是否全部完成（含校验）
bash ddd-workflow-status.sh <idea-dir> --check-all

# 查看整体状态
bash ddd-workflow-status.sh <idea-dir>

几个值得注意的设计决策：

纯 bash + awk，不依赖 jq/python
 — 兼容 macOS 默认环境（bash 3.2），降低部署门槛
flock 文件锁 + macOS mkdir 降级
 — 防止并发写入竞态（多个子代理可能同时更新状态）
非阻塞检查模式
 — 早期版本有 --wait-upstream 的阻塞轮询模式，但与 Claude Code Bash 120 秒超时不兼容，后来改为非阻塞检查，由主 agent 的 DAG 编排保证执行顺序
只注册需要开发的层
 — --init 时只传入评估为"需要开发"的层，不需要开发的层不注册，其 Phase 直接跳过
5.5 编排全景：‍从需求到代码的完整链路

把 Agent、Skill、Script 串起来，一个完整的后端开发流程是这样的：

整个链路中，Skill 负责流程编排（什么时候做什么），Agent 负责独立执行（在隔离的上下文中完成具体工作），Script 负责确定性操作（状态管理和契约校验）。三者各司其职，互不越权。

总结

这篇文章沿着两条主线展开：

第一条主线：‍为什么需要框架演进

纯提示词时代，模型已经掌握了我们的工程手段，75 分的生成质量触手可及，但缺少 plan with file、缺少阶段产物、缺少确定性工具。

轻量级 SDD 框架（Spec-Kit、OpenSpec）补上了这些短板，在千行级别的舒适区内，brainstorming → plan → task → impl 的链路配合一致性校验，7-8 成能一把过。

但当代码量级到了万行，上下文窗口成为硬瓶颈 —— 注意力稀释效应让模型在长上下文中失去重心，上下文压缩虽然有效但终究有极限。重量级框架（如 superpowers）通过子代理委派/集中式编排突破了这个瓶颈，几万行一把过也不是梦，但引入了新的风险：‍subagent 上下文完备性、并行冲突、TDD 重构机制的局限性，归根结底还是上下文在作怪。

当这个诉求放到企业级，集中 plan 成为瓶颈 —— 自研框架百八十个、几十上百领域成百上千的服务，plan 阶段需要的上下文大到单个 agent 会话都装不下。分层 plan 通过 DDD 四层拆分，将 plan 的上下文容量扩展了 3-4 倍，每层只关注自己需要的信息，从而缓解了企业级场景的规模瓶颈。

第二条主线：‍Agent、Skill、Script 的设计实践

Agent — 核心思路：‍结论先行、可操作性、分层详略、为下游消费设计

通过 Thinker/Worker 分离，让设计与实现各自拥有干净的上下文
通过权限分级（Thinker 用 disallowedTools:‍ Edit 硬性禁止越权，Worker 用 permissionMode:‍ acceptEdits 自动接受编辑），把软性 prompt 约束升级为硬性工具约束
通过 2-3 轮带证据的反思循环，防止模型"走过场"
通过 MISSION → TEMPLATE → CONTEXT → OUTPUT 四区块组装完备上下文，其中上游导出契约必须完整内联、上游设计文档全文改为按需 Read 加载，在信息完备性和上下文节省之间取得平衡

Skill — 核心思路：‍渐进加载

区分自主加载型（模型自动触发的流程编排）和任务型（显式调用的规范路由器，替代 Commands）
session-start hook 只注入几十 token 的触发索引而非全量规范
每一层 Agent 只在需要时才通过路由表按需加载对应的规范文件，确保每个环节的上下文精准不冗余

Script — 核心思路：‍固定逻辑走固定 Script

把状态管理、契约校验、进度查询这些确定性操作从模型推理中剥离出来，用 500+ 行的纯 bash 实现可审计的跨层一致性保障，避免模型"创造性地"解读状态
产出物的 frontmatter 中设计了 layer、order、status、depends_on 等标识字段，既作为 Script 校验的锚点，也让开发者能通过这些 ID 标识追踪每个文件在什么时候被加载、当前处于什么状态，提高整个执行过程的可观测性

三者各司其职：Skill 编排流程，Agent 独立执行，Script 兜底校验。

对于未来

写到这里，其实能感觉到一个很明显的趋势：‍我们现在做的所有事情 —— 分层 plan、契约校验、渐进加载 —— 本质上都是在帮模型"管理上下文"。但这件事，最终应该是模型自己来做的。

从编排到协作：‍Agent Teams

目前的框架是严格的自上而下编排：‍主 agent 派发任务，子代理执行完返回结果，子代理之间没有任何通信。这就导致一个问题 —— Worker 在实现过程中发现设计有缺陷时，它没有能力"向上反馈"，只能在自己的上下文里硬撑，产出一个带着妥协的实现。

我在尝试引入 agent teams 的模式。核心诉求是让 agent 之间可以通信：‍Worker 发现设计文档中某个接口签名在实际实现时会导致循环依赖，它应该能向 Thinker "开卡" —— 发起一个设计变更请求，Thinker 评估后修改设计文档并通知所有受影响的下游 agent。这不再是单向的委派，而是双向的协作。

但这也引入了新的复杂度：‍通信协议怎么定义？变更传播怎么控制范围？如果 Thinker 修改了 Domain 的导出契约，Infr 和 Application 的 Worker 是否需要暂停等待？这些问题目前还没有很好的答案，但方向是确定的 —— 从"编排"走向"协作"，从"主从"走向"对等"。

从单项目到跨仓自治：‍code 平台的终局

再往远看一步。假设上下文管理和生成质量都不再是瓶颈 —— 模型的上下文窗口足够大，或者模型自己学会了比我们更好的上下文管理策略 —— 那当前这种"人搭框架、框架编排 agent"的模式就是过渡态。

终局应该是：‍code 平台拿到一个业务需求后，自己分析这个需求涉及哪些代码仓、哪些服务，自己决定在每个仓库上做什么变更，自己处理跨服务的接口协商和版本兼容，自己跑测试、自己发 PR。开发者的角色从"编排 AI 干活"变成"审核 AI 的方案和产出"。

这不是科幻。当前框架里的 Decision-Maker 已经在做"需求 → 层级评估 → 分发"这件事了，只是范围限定在单个项目的 DDD 四层内。把这个范围扩大到企业级的多仓多服务，技术上是同一套思路的延伸 —— 更大的 DAG、更多的 agent、更复杂的契约体系。真正的挑战不在技术，而在信任：‍你敢让 AI 自己决定拆不拆服务、改不改接口吗？

AI 视角下的架构认知差异

这就引出一个更深层的问题。我们现在的微服务架构、领域划分、服务拆分策略，很大程度上是为人的认知能力设计的。一个团队 5-8 个人，能同时理解和维护的代码规模是有限的，所以我们拆服务、划边界、定契约 —— 本质上是在用架构手段降低单个人需要理解的复杂度。

但 AI 的认知上限和人不同。人眼中的"大单体" —— 几十万行代码、上百个模块 —— 对 AI 来说可能只是一个中等规模的上下文。人需要拆成 20 个微服务才能管理的系统，AI 可能作为 3 个服务就够了，甚至保持单体反而更高效 —— 因为跨服务调用的复杂度（序列化、网络延迟、分布式事务）对 AI 来说也是额外的上下文负担。

当然，微服务的拆分不只是认知问题，还有性能隔离、安全边界、独立部署、团队自治等客观诉求。但如果"团队自治"这个因素被 AI 大幅弱化（因为 AI 不需要按团队划分认知边界），那最优的架构形态一定会发生变化。未来的架构决策，可能不再是"人能理解多少"，而是"AI 在什么粒度上生成和维护代码最高效"。

框架的宿命：‍为了消灭自己而存在

说到底，这篇文章里描述的所有框架和工程手段，都是当前模型能力的"补丁"。上下文窗口不够大，我们就分层 plan；模型不够听话，我们就用工具权限硬约束；契约可能不一致，我们就写 500 行 bash 做校验。这些都是在模型能力和工程需求之间的 gap 里填东西。

当模型能力持续提升 —— 更大的有效上下文、更强的指令遵循、更好的自我纠错 —— 这些补丁会一个个变得不再必要。渐进加载？模型自己会管理注意力分配。契约校验？模型自己能保证跨层一致性。权限分级？模型自己知道什么该做什么不该做。

但在那一天到来之前，这些框架就是当下最务实的解法。而且搭建框架的过程本身也有价值 —— 它迫使你把"AI 辅助编程"这件事想清楚：‍哪些是确定性的、哪些是需要推理的、哪些信息在什么时候该出现。这些思考，即使框架本身过时了，认知也不会过时。

归根结底，AI 辅助编程的核心变量是上下文管理。所有框架演进的本质，都是在回答同一个问题：如何在有限的上下文窗口中，让模型看到最该看到的信息，做出最该做出的决策。
