## Page 006

现有方案为什么不够？

副标题：RAG、向量数据库、长上下文——都不是“记忆”

表格：

| 方案 | 做了什么 | 缺了什么 |
| --- | --- | --- |
| RAG | 检索外部文档片段注入上下文 | 无状态——不记得上次检索了什么；无法关联多次交互；无记忆管理 |
| 向量数据库 | 提供高效的相似度检索 | 只是存储层——没有理解、没有推理、没有遗忘与压缩 |
| 长上下文窗口（2M token） | 一次性塞入更多信息 | 成本爆炸（$0.5/次）、延迟高（分钟级）、无法持久化、fine-tune难度高 |
| Mem0 / 会话记忆 | 记住对话中的关键事实 | 仅纯文本会话、无多模态、无企业级特性、无大规模验证 |

底部强调：

> 这些方案解决的是“数据接入”问题，不是“记忆”问题。

公式：

记忆 = 理解 + 存储 + 组织 + 推理 + 遗忘 + 演进

类型包括：

痛点1：多模态数据理解的“最后一公里”

左侧问题描述：OCR + 通用模型在复杂业务场景中频繁失效：

- 复杂业务表格（合并单元格、转置表、嵌套子表）
- 手写批注、盖章遮挡
- 会议录音中的情绪/语气识别

我们踩过的坑（表格）：

| 场景 | 通用方案准确率 | 问题 |
| --- | --- | --- |
| 多子表Excel | ~55% | 无法识别一个worksheet里的多个逻辑表 |
| 跨页PDF表格 | ~40% | 表头丢失、行错位 |
| 中英文混排发票 | ~60% | 字段映射错误 |
| 财务合并报表 | ~45% | 合并单元格语义丢失 |

右侧我们的解法：

1. 专用提取Pipeline而非通用VLM：Excel流程 + PDF流程
2. 语序恢复：结合语义的OCR后处理，修复布局识别导致的语序错误
3. 视觉+逻辑双重验证：提取结果与原始文档视觉比对 + 逻辑一致性校验
4. Skills自定义：客户可以为特定文档类型定义专属提取规则

底部效果：

办公场景准确率从行业平均60-70% → 99.8% 召回率

## Page 024

痛点2：10万亿级记录下的检索成本失控

副标题：维度2——检索成本失控

引用：向量检索在千万级规模很开场体验，到了亿级、百亿级，问题直接恶化。

上方表格/条形图：数据（某客户实测）

- 记录规模 100万：检索延迟 10ms，精度(Recall) 95%
- 1亿：80ms，95%
- 100亿：600ms，95%
- 10万亿：timeout（红色条）

右侧指标：

- <200ms：10万亿级记录检索延迟（vs 2分钟timeout）
- 95%+：精度保持
- 30x：成本降低

我们的解法——混合检索 + 记忆分层：

1. 多策略召回：关键词索引（精确匹配）+ 向量索引（语义匹配）+ 图索引（关系匹配），三路并行取并集
2. 记忆分层存储：热记忆 / 温记忆 / 冷记忆
3. 自适应路由：分析查询意图，一层层检索，避免全量扫描式灾难
4. 查询路由：QueryAgent根据问题类型自动选择检索路径（不需要每次都查全量）

- Instance：世界观设定（所有NPC共享）
- Agent：NPC角色记忆（每个NPC独立）
- Session：玩家本次对话上下文（会话结束后压缩入Agent记忆）
