## Page 019

```text
    行业陷阱：平庸化的知识库RAG
行业现状：       行业平台都在推     ”文件上传 -> 自动切片 -> 向量检索 -> 回答”。

           这种简单的 RAG 模式对于结构化差、逻辑性强、多跳推理（Multi-hop reasoning）的场景几乎无效。用户
主要问题：      上传一堆文档，以为有了知识库就万事大吉，结果 Agent 经常答非所问，或者只能做简单的片段拼接。


后果：      用户误以为是大模型不行，其实是检索架构太浅。平台为了易用性，牺牲了检索的深度和精度。


破解之道：从“被动检索”转向“主动构建与推理支持”
1   超越vector search 平台必须原生支持 Graph RAG（知识图谱增强）和 Hybrid Search（混合检索）、RAG
                 pipeline定制扩展。不仅仅是存文本，更要自动提取实体关系，构建局部知识图谱。
               在检索前，强制加入一个“Query Decomposition”层，将复杂问题拆解为多个子问题，分别检
2   查询重写与分解    索后再综合。用户的问题比较模糊，需要对其进行改写或者扩写，以提高召回准确率。

               平台不能只做“上传即忘”的仓库，而要提供数据质量评分和缺失知识发现、知识扩写 等
3   数据治理前置     功能。告诉用户：“你的知识库里缺少关于 X 的关键信息，导致无法回答 Y 类问题。”

4   多模态的支持      不能局限于文本态的知识库，更需要多模态的能力，图片、语音、视频、时空轨迹
```

## Page 020

```text
  行业陷阱 ：“乐高积木”幻觉 —— 复杂系统的脆弱性

行业现状： 行业平台的核心逻辑是“拖拽节点”。用户认为只要把“搜索 -> 总结 -> 判断 -> 输出”连起来，就能得到一个强
          大的 Agent。

  问题：     现实世界的业务逻辑（如骑手配送）充满了不确定性、模糊性和长尾异常。基于固定流程图（DAG）的 Agent，一旦遇
          到未在流程中定义的分支（例如：用户说了一句模棱两可的话，或者外部 API 返回了非标准格式），整个链路就会断裂
          或产生幻觉。

  后果：    生产了大量“玩具级”应用，只能处理 理想情况，生产环境效果一般


突破之道：
  1   不再依赖硬编码流程       平台不应只支持固定的 Workflow，而应内置 ReAct (Reasoning + Acting) 或 Plan-and-
                      Solve 的运行时引擎。

  2   引入“反思机制”          Agent 在执行每一步后，必须有一个自我评估环节（Self-Correction）。如果置信度低于阈
      (Reflection Loop) 值，自动触发重试或切换策略，而不是直接报错。

  3   平台升级方向          目前的推拉拽工作流，只是中间状态，nl2workflow、self-improving workflow是未来；


心得：不要让用户画地图，要让 Agent 自己探路。
```
