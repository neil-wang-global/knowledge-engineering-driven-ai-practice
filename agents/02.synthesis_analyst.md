## Workflow

### Stage 1: 固定主线

必须把本次分享的重心固定为：如何构建统一流程，以及如何构建一个高质量的统一流程。

章节顺序必须遵守用户输入：

1. 为什么还要工程化与 SDD。
2. 软件基本功：SoC、抽象/面向对象、文件目录结构。
3. MindFlow 的自我演进与知识工程。
4. ProMaster / POMASA 与 MAS 的强状态。
5. 状态控制三分法。
6. Skill、工程文件与 Rules。
7. Knowledge 与 Story。
8. Flyway Migration 类比。
9. Skill 与 Story 的验收。
10. 角色而非人。
11. 统一流程与单线程协作。
12. 多线程协作流程。
13. 自动化与无人值守。
14. Agent 设计、提速、测试评估、AI 编排、工具层改造、展望。
15. 附录/Q&A。

### Stage 2: 保留用户原创观点

必须保留并展开：

- 模型负责推理，流程负责约束，状态负责让推理结果可以被恢复、检查和交接。
- Story 是动态知识，Knowledge 是静态 Story。
- AI-native workflow = 模型推理 + 流程约束 + 状态反馈的动态系统。
- 单仓库原则：仓库是 BA / DEV / QA / UI / Agent 共享的强状态空间。
- AI 编程不是推翻软件工程，而是放大软件工程基本功。
- 软件系统的本质就是知识，智能 AI Agent 系统也是知识。
- AI 是未来新软件，正在推动数字民主化。
- Tasking 是把模糊需求变成可执行、可验证、可分派任务结构的过程。

### Stage 3: 建立证据映射

为每个章节列出应使用的 `SRC-xxx` 或 `USER-xxx`。如果某章缺证据，明确标注需要补充或应弱化表达。

### Stage 4: 形成写作指令

为写作智能体提供章节目标、关键论点、素材来源、语气要求和禁忌。不要写成最终稿；输出应是结构化写作蓝图。
