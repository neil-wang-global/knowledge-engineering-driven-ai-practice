知识工程驱动的 AI 实践

从 AI Coding 到可运行、可协作、可演进的 AI 工作系统。

王唯力 Neil Wang

今天只讲一条主线

如何构建统一流程，以及如何构建一个高质量的统一流程。

每个概念都要落回流程位置：输入是什么，输出是什么，状态在哪里，谁负责，如何验收，失败如何回流，知识如何沉淀。

第 0 章：问题变了

现在的风险经常不是 AI 不会写，而是 AI 太快开始写。

需求没澄清，边界没想清，验证没设计，它已经开始补默认值、写方案、生成代码。

过早收束：流畅感掩盖问题空间

模糊需求意图、用户路径、验收标准还没展开。
单一路径模型快速选择一个看起来合理的方向。
默认假设缺失信息被自动补齐，但没有显式确认。
快速实现代码出现得很快，风险也被带进系统。
隐性债务异常、权限、状态、验收被延后暴露。

这页用于解释为什么“能生成”不等于“能交付”。

为什么中端 / 低端模型更需要结构

强模型 + 明确任务

直接交互通常够用，外部结构可以轻一点。

强模型 + 开放任务

需要前置边界、检查点和验收，防止越做越偏。

中低端模型 + 明确任务

用清晰输入、固定格式、测试命令兜住输出质量。

中低端模型 + 开放任务

SDD、MAS、Rules、状态文件都要增强。

模型能力 ↑
任务开放度 →

方法论没有消失，价值迁移了

过去：能力补丁替模型拆任务、补步骤、补记忆、补稳定性。方法论主要是让模型“会做”。
现在：治理工具定义边界、固定检查点、控制权限、沉淀状态、保障验收。方法论主要是让系统“稳定做对”。

模型越强，越不能只问“它会不会做”，还要问“系统能不能稳定地做对”。

SDD / MAS / Rules 分别解决什么

SDD

把“想清楚”和“开始干”分开。显化 what、why、guardrails、验收标准和关键边界。

MAS

把复杂任务拆成角色、上下文、权限、阶段、产物和验收，让协作可审计。

Rules / Hooks

把必须发生的动作交给确定性控制：检查、权限、引用、测试、完成声明。

三者不是互斥工具，而是不同层次的流程控制。

第 1 章：软件基本功

SoCStory、ADR、Spec、Plan、State、Review、Eval 各司其职。材料不混放，职责才清楚。
抽象 / OOPSkill、Rule、Capability、Agent Blueprint 是知识对象。对象边界不清，Agent 就会误用。
目录结构不是收纳方式，而是 Agent 可导航的知识地址空间。路径本身就是提示和协议。

软件基本功迁移到 AI 文件工程

传统工程

模块、接口、测试、目录结构，让人能协作维护。

AI 文件工程

Story、Spec、Plan、State、Skill、Eval，让 Agent 能理解、执行、验收。

共同目标

降低耦合、减少歧义、让变化有位置、让错误可定位。

嵌套知识对象：不要平铺材料

Repository团队共享强状态空间，承载当前事实和流程入口。
Story业务意图、用户路径、验收标准，描述一次知识变化。
ADR / Spec决策、接口、边界、不可做事项，固定关键判断。
Plan / State执行路径、当前进度、失败回流，让任务可恢复。
Test / Eval / Skill验证证据、复用能力、知识回写，让经验可复用。

事实分叉：Agent 最怕多个权威

复制式文档

同一规则散落在 Story、Plan、Review。改一处，漏两处，下次 Agent 不知道信谁。

引用式知识

事实有唯一权威位置，其他文件通过 ID / link / reference 依赖。

工程效果

Agent 能判断当前状态，团队能追踪变化来源，Review 能定位责任边界。

第 2 章：MindFlow 的自我演进

GitHub: https://github.com/neil-wang-global/MindFlow · Paper: https://zenodo.org/records/18977748

不要把所有知识揉成一个巨大 prompt

巨大 prompt

长期原则、项目知识、任务状态、历史材料混在一起，权威关系不清。

分层知识

长期原则、能力、计划、状态、反思各有位置，按阶段加载。

自我演进

任务状态不丢，能力可复用，反思能回写，系统沿着已有知识继续推进。

知识工程是什么

不是把提示词写长，而是把人类判断组织成 AI 可消费的知识资产。

问题导向定义要解决什么问题，以及为什么值得解决。
价值框架选择分析视角、约束和判断标准。
信息储备积累、筛选、组织高质量材料。
理论方法决定研究路径、拆解方法和执行范式。
表现形式根据场景选择报告、代码、演示、流程或系统资产。

软件系统的本质就是知识

传统软件代码固化业务规则，文档解释意图，测试描述期望行为，规则沉淀团队经验。
Agent 系统上下文、状态、工具、反馈、Skill、eval case，同样都是知识的表达。

知识工程关注的不是一次回答，而是系统如何持续积累、复用和校正知识。

第 3 章：POMASA 与 MAS 的强状态

MAS 的关键不是“几个 Agent 聊天”，而是围绕状态推进的编排系统。

先定义状态和产物，再定义 Agent；否则多智能体只会变成聊天接力。

POMASA 的工程信号

Blueprint / Pattern把角色、阶段、输入输出声明出来，让协作结构可复用。
Filesystem Data Bus文件系统承载多角色产物流转，降低对聊天上下文的依赖。
Verifiable Data Lineage产物来源、验收、回流可追踪，出错时能定位阶段。

GitHub: https://github.com/eXtremeProgramming-cn/pomasa

强状态不是“全都写下来”

关键产物

有稳定位置，有命名，有所有权。下游知道读什么。

关键判断

有来源，有理由，有可追溯记录。Review 知道为什么。

关键验收

有证据，有结果，有失败回流。系统知道是否完成。

聊天接力 vs 强状态流水线

聊天接力

A 总结给 B，B 总结给 C。信息持续压缩，责任边界模糊。

强状态流水线

每一步产出文件，每一步有验收，下游读取明确产物。

关键差异

能恢复、能审计、能定位错误来自哪个阶段。

第 4 章：状态控制三分法

强约束固定Rules、Hooks、必读文件、测试、提交检查。适合安全、权限、验收声明。
建议式调度Skill、推荐流程、可选 checklist。适合可复用能力和经验方法。
完全 AI 控制探索、低风险实验、一次性任务。适合开放发散，不适合关键交付兜底。

可靠性与灵活性的张力

强约束：可靠↔建议式：平衡↔AI 自主：灵活

高质量流程的关键不是选一种，而是组合使用。

必须发生的动作交给强约束；可复用方法交给 Skill；开放探索交给模型自由判断。

第 5 章：Skill、工程文件与 Rules

Skill可复用能力；触发依赖模型判断，存在漏触发和误触发。
工程文件结构化上下文；处在流程规定位置，比 Skill 更确定。
RulesHook 式强约束；稳定，但自由度低，适合兜底动作。

三者都是可执行知识，但进入流程的方式不同。

Skill 构建及评定：先变成任务

一个 Skill 不是“写得很完整”就算成熟，它要能被一组任务检验。

Task独立测试用例：输入、上下文、工具、成功标准都要明确。
真实来源从手动测试、线上失败、用户反馈、Review 问题中提取任务。
平衡样本既测应该触发，也测不该触发，避免 Skill 过度调用。

Skill 评定：看一次 Trial 的完整轨迹

输入任务是否清楚，参考答案或成功标准是否可判定。
触发Agent 是否在正确时机加载 Skill，是否漏触发。
执行是否读取必要文件、调用必要工具、产出中间状态。
Transcript记录推理、工具调用、错误和修正路径，用于审查。
Outcome最终环境是否真的满足任务，而不是只看模型自述。

Outcome 优先：不要只看 Agent 说了什么

自述完成

“我已经完成了”“我检查过了”。这是信号，不是证据。

轨迹证据

读了什么文件、用了什么工具、跑了什么检查、遇到什么失败。

最终状态

测试结果、文件 diff、引用链、浏览器状态、数据库状态或交付物状态。

Agent eval 的核心提醒：评估最终状态，必要时再回看路径。

Grader：评分器要组合使用

代码评分器单元测试、静态分析、格式检查、状态检查。快、便宜、客观、可复现。
LLM 评分器用于开放判断、质量标准、自然语言断言，但要和人工校准。
人工评分器用于关键场景、价值判断和抽样校准。贵，但能提供黄金标准。

尽可能用确定性评分器兜住客观部分，把模型评分留给真正需要语义判断的部分。

Evaluation Suite：让 Skill 能回归

能力评估

问“这个 Skill 擅长什么，还能提升到哪里”。任务要有挑战性，不能太快饱和。

回归评估

问“以前能做对的事，现在还做不做得对”。通过率应接近 100%。

失败回流

真实失败进入任务集，防止同类错误复发。失败不是结束，是 eval case 的来源。

三类控制要组合使用

Skill

提供可复用能力，但触发依赖模型判断，所以要用任务集评估触发质量。

工程文件

提供稳定上下文和流程位置，但需要流程规定什么时候读取、写回哪里。

Rules / Hooks

提供确定性兜底，但太宽会打断，太窄会漏风险，需要持续校准。

第 6 章：Knowledge 与 Story

Story 是动态知识，Knowledge 是静态 Story。

User journey 是团队发现、验证、固化知识的过程。

Story 是一次知识变化

为什么变用户为什么要这个变化，旧系统哪里不满足。
怎么变新的业务路径、边界条件、规则和设计选择。
如何证明验收标准、测试、trace、outcome 和知识回写。

回写闭环

Story→代码文档测试Eval Case→Knowledge
SOUL.md指导Skill 进化→流程规则→下一次执行

完成一个需求，不只是代码变了；长期原则、Skill 和当前知识状态都要一起演进。

Story 完成后，知识必须进入当前状态

只改代码下次仍要重新解释需求，失败模式无法复用，Agent 不知道历史判断。
代码 + Knowledge文档、测试、Skill、eval case 一起更新，资产增长，流程越跑越稳。

第 9 章：角色而非人

BA业务意图、用户路径、验收标准。
DEV设计、实现、工程风险和技术边界。
QA / UI可测性、用户体验、异常路径和真实使用场景。
Agent检索、执行、验证、总结、回写。

角色是思考帽，不是固定人头；一个人或一个 Agent 都可能承担多个角色动作。

单仓库原则

仓库是 BA / DEV / QA / UI / Agent 共享的强状态空间。

需求、设计、实现、测试、验收和回写都在同一个可审计空间里发生，减少部门墙和上下文传递损耗。

user-journey-001/一条完整用户路径：从触发、操作、反馈到业务结果。
story-001/story.md用户故事、AC、边界条件、异常路径和验收口径。
story-001/tests.md不放测试本体，而是引用 knowledge/qas/ 中对应测试资产。
story-001/ui.md可选文件：如果有 UI，则引用 knowledge/uis/ 中的设计资产。

Story 内部：plans/ 是执行控制台

story.md业务意图、AC、边界条件，是执行的最小需求单元。
tests.md引用 knowledge/qas/，说明对应测试、缺口和通过状态。
ui.md可选引用 knowledge/uis/，没有 UI 就不强行生成设计资产。
plans/progress.md记录当前状态、阻塞点、下一步，支持断点续跑。
plans/plan.md总计划：任务拆解、风险、执行顺序和验收路径。
frontend-plan.md前端 Agent 重点读取：页面、组件、状态和交互。
backend-plan.md后端 Agent 重点读取：接口、领域逻辑、数据和权限。
api-contract.md前后端协作契约：请求、响应、错误码、字段含义。
qa.md验收证据、测试结果、截图、录屏和人工确认。
review.md代码 Review、设计 Review、测试 Review 的结论和遗留项。
knowledge 回写稳定经验进入 knowledge/，不稳定判断继续回到 references/wips/。
小闭环一个 Story 自带计划、状态、契约、QA、Review 和回写位置。

`plans/` 不只是计划文件夹，而是一个 Story 的调度台：谁读什么、做到哪一步、怎么恢复、如何验收，都写在这里。

Agent 编排：入口、规则、技能、知识分层

CLAUDE.md项目入口。Agent 进项目必须从这里理解启动方式、边界和基本约束。
.claude/settings.jsonHook 配置。把“必须走 CLAUDE.md”变成可执行门禁。
.claude/rules/项目级强约束。这里只保留关键规则，例如 must-read-claude.md。
agents/orchestrator.md流程编排器：定义角色顺序、读写边界、进入下一步的条件。
agents/skills/BA、UI、Frontend、Backend、QA、Reviewer 等角色技能。
knowledge/soul/SOUL.md长期方向总纲，引用 soul/ 下持续增长的原则和演进文件。

最终形态：仓库管理天然适合权限控制

documents 项目知识性东西全部放这里：references、products、story、plans、agents、knowledge。
frontend / backend 项目代码仓库回到纯粹：前端项目只有前端代码，后端项目只有后端代码。
按角色读取前端读 Story 中 plans/frontend-plan.md 和 api-contract.md；后端读 backend-plan.md 和 api-contract.md。
商务资料隔离标书、合同等敏感资料可以放在另一个项目，只给需要的人开放权限。
权限天然分层不同级别用户能 clone 的项目不同，在本地 group 目录下可工作的范围也自然不同。

多仓库不是把知识打散，而是把代码、知识、商务材料按权限边界拆开；统一流程仍然由 documents 项目承载。

从单线程到低人工干预

单线程闭环一个需求先能从输入、实现、测试到验收稳定跑通。
Worktree 并行把统一流程复制到多个隔离增量，不把并行当成自由发挥。
长时间运行用 feature list、progress file、clean state 保持状态连续。
低人工干预自动检查、可观测、人工 gate 和失败回流共同兜底。

这三章讲的是一件事：先把闭环做稳，再把稳定闭环复制到并行和长时间运行里。

多线程不只是更多 Agent 同时写代码

Feature List先列出功能增量，保持范围清楚。
Story拆成可验收、可回写的用户故事。
解依赖识别接口、数据、顺序和共享文件。
并行推进无依赖 Story 进入隔离 worktree。
完成 / 下一项验收、合并、回写，再推进下一批。
BA 技能拆 Scope、AC、业务边界和依赖。
QA 技能生成主路径、异常路径和测试矩阵。
DEV 技能按 Spec、Plan、Test 实现最小增量。
UI 技能确认用户路径、交互状态和可用性。
验收技能Review、E2E、trace、outcome 统一判断。
回写技能把经验写回 Knowledge、Skill 和 Eval case。

低人工干预：降低沟通损耗，不是取消治理

人月神话讲的是沟通损耗；结构化知识让 AI 协作的沟通损耗大幅下降。

当 Agent 共享同一套知识资产、状态文件和验收标准，软件迭代速度的跃迁来自协作结构变化，而不是 token 生成更快。

任务边界避免 Agent 自己扩大范围。
Worktree隔离并行执行状态。
Progress File跨 session 保持连续性。
自动测试用确定性工具给反馈。
可观测看见路径、错误和修复。
人工 Gate高风险处停下来确认。

长时间运行需要强状态

Feature List知道要完成哪些增量，避免 Agent 自己扩大范围。
Progress File知道现在做到哪里，下一次接手不用重新猜。
Clean State知道从哪里安全恢复，避免脏状态污染判断。

无人值守的前提不是“没人管”，而是状态足够清楚、验证足够自动、风险点有人工 gate。

提速来自哪里

上下文损耗下降强状态空间减少重复解释和交接误差。
反馈更快测试、评估、浏览器验证前移，失败更早暴露。
资产持续增长失败变成 eval case，经验变成 Skill，判断变成 Knowledge。

工具层改造

人类体验页面、说明、宽松输入、可读输出。目标是让人看懂、愿意用。
智能体体验JSON、schema、字段裁剪、稳定错误码、权限边界、输入防御。目标是让 Agent 稳定调用。

Agent 是不受信任的操作员

为什么不能默认信任

Agent 没有恶意，但会误解上下文、过早收束、遗漏边界，也会把看起来合理的动作当成正确动作。

人类操作员

会停下来问、能读空气、知道组织禁区，很多约束来自经验和责任。

Agent 操作员

只能依赖上下文、工具协议和反馈信号；如果工具层没有边界，它就会把风险带进系统。

纵深防御：不要靠模型记住安全边界

最小权限默认只给完成任务所需权限，写入、删除、发布、生产操作都要显式授权。Permission
结构化输入工具输入用 schema、枚举和字段裁剪，减少 prompt 注入和参数误解。JSONSchema
副作用隔离高风险动作先预览、再确认、再执行，执行后留下可回看的 trace。ConfirmTrace

Agent 友好的工具协议

稳定错误码

不要只返回“失败了”，要让 Agent 知道是权限、校验、网络、状态冲突还是服务端错误。

可审计轨迹

记录读了什么、改了什么、调用了什么工具、为什么停下，Review 才能判断是否可信。

可恢复状态

失败后能回到安全状态，下一轮 Agent 能从状态文件和工具结果继续，而不是重新猜。

工具层的目标不是限制 AI，而是把不确定性关在可观察、可回滚、可验收的边界内。

最后一个议题：先选形态，再谈 Agent

Workflow

路径预定义，LLM 和工具按固定代码路径执行。适合可拆解、验收明确、需要一致性的任务。

Agent

模型动态决定步骤和工具。适合开放、多轮、路径难以预判的问题。

选择原则

先用最简单方案，只有当复杂度明确改善 outcome 时才升级。

建立有效代理不是追求最复杂，而是为问题选择合适的复杂度。

Prompt Chaining：把大任务切成固定链路

Step 1先产出中间结果，例如大纲、方案、草稿。
Gate用程序或模型检查中间结果是否达标。Check
Step 2下一步只处理上一步通过后的产物。
Outcome用更小、更清晰的任务换取更高准确率。

适合能清晰拆成固定子任务的问题；优势是稳，代价是多轮调用带来延迟。

Routing：先分类，再进入专门流程

输入→分类器→流程 A流程 B流程 C→结果
解决问题不同类型输入需要不同提示、工具、模型或权限。
优势SoC 清晰，避免一个提示同时优化所有场景。
风险分类错了，后面的专业流程会越做越偏。

Parallelization：并行处理，最后聚合

任务→子任务 1子任务 2子任务 3→聚合 / 投票

Sectioning

把独立子任务并行处理，适合多维检查、多模块分析。

Voting

同一任务跑多次或多视角判断，适合安全审查、质量评估。

代价

需要处理结果冲突、聚合规则和成本上升。

Orchestrator-workers：动态拆解和委派

Orchestrator先理解任务，再决定要拆哪些子任务。
Workers多个 worker 分别处理文件、模块、资料或检查项。
Synthesis主控汇总结果，处理冲突，决定下一步。
Verification用测试、Review、trace 判断是否真的完成。

适合跨文件编码和复杂搜索；风险是 worker 上下文不完整、并行冲突和主控漏验收。

Evaluator-optimizer：生成、评价、再优化

生成器→草稿 / 代码→评价器→反馈↺优化
适用前提评价标准清楚，反馈能稳定提升结果。
适合场景写作润色、翻译、复杂搜索、代码修复。
风险评价器标准不稳，会把系统带进反复修改。

Autonomous Agent：开放任务才需要高自主

目标人给出任务或通过对话澄清目标。
环境反馈Agent 通过工具结果、代码执行、测试结果获得 ground truth。
检查点遇到阻塞或高风险判断时暂停请求人类反馈。
停止条件完成条件、最大轮次、成本上限或人工 gate。

自主性越高，越要配 sandbox、guardrails、测试和人工检查点。

AI 辅助编程：代码规模决定方案强度

百行级直接 Vibe Coding 或单次提示通常够用，重点是快速反馈。Prompt
千行级轻量 SDD 更合适：澄清、Plan、Task、Impl，把阶段产物留下来。OpenSpecSpecKit
万行级单会话上下文会滚雪球，需要子代理委派隔离执行过程。Sub-agentSuperpowers

生成速度的上限不是单纯由模型决定，而由上下文密度、任务拆分、验证反馈和状态隔离共同决定。

百行级：直接生成，快速反馈

明确小需求→单次提示 / Vibe Coding→运行检查→小步修正
为什么可行上下文短，模型容易抓住重心。
检查重点lint、局部测试、人工快速 Review。
不要过度工程化小任务不必先搭复杂框架。

千行级：轻量 SDD 留下阶段产物

Brainstorming澄清需求，确认模型对任务的理解。
Plan把实现方案写下来，避免边写边猜。
Task拆成可执行步骤，降低单次生成复杂度。
Impl + Check逐步实现，并用测试和一致性检查兜底。

这类方案适合日常需求开发：流程不重，但能留下可恢复、可检查的阶段产物。

万行级：子代理委派突破上下文滚雪球

主 Agent拆 TaskSub-agent ASub-agent BSub-agent C→汇总 / 验收

优势

每个子代理上下文更干净，执行过程不挤占主会话。

风险

委派 prompt 可能不完整，并行工作可能冲突。

兜底

用测试、Review、Playwright、状态文件和人工 gate 验收。

选择方法：先问四个问题

路径能否预定义？能预定义就优先 workflow；不能预定义才考虑更自主的 agent。
上下文会不会爆？会爆就拆 task、隔离会话、分层处理，而不是硬撑一个窗口。
结果能否验证？测试、lint、eval、review 越清楚，越适合自动化推进。
错误代价多高？代价高就加强 gate、权限、trace 和人工确认。

最后带走一句话

AI-native workflow 是模型推理、流程约束和状态反馈的动态系统。

模型负责推理，流程负责约束，状态负责让推理结果可以被恢复、检查和交接。
