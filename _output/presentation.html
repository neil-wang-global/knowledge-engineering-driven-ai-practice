AI Work System Speech

知识工程驱动的 AI 实践

从 AI Coding 到可运行、可协作、可演进的 AI 工作系统。

60-90 分钟演讲 · 技术团队 / 客户技术听众

今天只讲一条主线

如何构建统一流程，以及如何构建一个高质量的统一流程。

所有概念最终都要回到：输入、输出、状态、角色、检查点、验收和回写。

第 0 章：问题变了

现在的风险经常不是 AI 不会写，而是 AI 太快开始写。

需求没澄清，边界没想清，验证没设计，它已经开始补默认值、写方案、生成代码。

过早收束：流畅感掩盖问题空间

模糊需求意图、用户路径、验收标准还没展开。
单一路径模型快速选择一个看起来合理的方向。
默认假设缺失信息被自动补齐，但没有显式确认。
快速实现代码出现得很快，风险也被带进系统。
隐性债务异常、权限、状态、验收被延后暴露。

为什么中端 / 低端模型更需要结构

强模型 + 明确任务

直接交互通常够用，外部结构可以轻一点。

强模型 + 开放任务

需要前置边界、检查点和验收，防止越做越偏。

中低端模型 + 明确任务

用清晰输入、固定格式、测试命令兜住输出质量。

中低端模型 + 开放任务

SDD、MAS、Rules、状态文件都要增强。

模型能力 ↑
任务开放度 →

方法论没有消失，价值迁移了

过去：能力补丁替模型拆任务、补步骤、补记忆、补稳定性。
现在：治理工具定义边界、固定检查点、控制权限、沉淀状态、保障验收。

模型越强，越不能只问“它会不会做”，还要问“系统能不能稳定地做对”。

SDD / MAS / Rules 分别解决什么

SDD

把“想清楚”和“开始干”分开。显化 what、why、guardrails、验收标准和关键边界。

MAS

把复杂任务拆成角色、上下文、权限、阶段、产物和验收，让协作可审计。

Rules / Hooks

把必须发生的动作交给确定性控制：检查、权限、引用、测试、完成声明。

AI-native workflow

模型推理生成方案、代码、解释、判断。
流程约束先想清楚，固定检查点，验收显化。
状态反馈可恢复、可检查、可交接、可回写。

第 1 章：软件基本功

SoCStory、ADR、Spec、Plan、State、Review、Eval 各司其职。
抽象 / OOPSkill、Rule、Capability、Agent Blueprint 是知识对象。
目录结构不是收纳方式，而是 Agent 可导航的知识地址空间。

上下文管理不是越多越好

给模型低噪声、可处理、可验证的上下文。
确定性工具处理格式 / 测试 / 检查模型推理 / 权衡 / 生成

第 2 章：MindFlow 的自我演进

Soul长期原则与方向
Learning外部知识与经验输入
Capability可复用能力单元
Plan / Task State执行路径与当前状态
Reflection复盘、沉淀、更新

不要把所有知识揉成一个巨大 prompt

  • 长期原则、项目知识、能力、状态、反思要分层。
  • Lazy loading 是流程设计，不只是省 token。
  • 任务状态让系统可恢复、可交接、可复盘。
  • 知识空间不是材料堆砌，而是可被 Agent 操作的结构。

软件系统的本质就是知识

传统软件代码、文档、测试、规则，是知识的固化。
Agent 系统上下文、状态、工具、反馈，也是知识的表达。

第 3 章：POMASA 与 MAS 的强状态

MAS 的关键不是“几个 Agent 聊天”，而是围绕状态推进的编排系统。

谁读取谁产出放在哪里谁验收失败如何回流

POMASA 的工程信号

Blueprint / Pattern可声明、可复用的协作结构。
Filesystem Data Bus文件系统承载多角色产物流转。
Verifiable Data Lineage产物来源、验收、回流可追踪。

强状态不是“全都写下来”

关键产物

有稳定位置，有命名，有所有权。

关键判断

有来源，有理由，有可追溯记录。

关键验收

有证据，有结果，有失败回流。

第 4 章：状态控制三分法

强约束固定Rules、Hooks、必读文件、测试、提交检查。
建议式调度Skill、推荐流程、可选 checklist。
完全 AI 控制探索、低风险实验、一次性任务。

可靠性与灵活性的张力

强约束：可靠↔建议式：平衡↔AI 自主：灵活

高质量流程的关键不是选一种，而是组合使用。

第 5 章：Skill、工程文件与 Rules

Skill可复用能力；触发依赖模型判断，存在随机性。
工程文件结构化上下文；处在流程规定位置，更确定。
RulesHook 式强约束；稳定，但自由度低。

成熟 Skill 像函数

触发输入前置假设步骤输出权限副作用验证下游消费

第 6 章：Knowledge 与 Story

Story 是动态知识，Knowledge 是静态 Story。

User journey 是团队发现、验证、固化知识的过程。

第 7 章：Flyway 类比

Story = Migration带着时间性、变化意图和业务增量。
Knowledge = 当前状态所有已验证增量执行后的知识系统。

回写闭环

Story→代码文档测试SkillEval Case→Knowledge

第 8 章：验收 Skill 与 Story

Skill 验收触发、输入、步骤、输出、权限、副作用、验证、下游消费。
Story 验收业务意图、AC、依赖、边界、可测试性、知识沉淀。

不要只相信“我完成了”

第 9 章：角色而非人

BA业务意图与验收标准。
DEV设计、实现、工程风险。
QA / UI可测性、业务路径、用户体验。
Agent检索、执行、验证、总结。

单仓库原则

仓库是 BA / DEV / QA / UI / Agent 共享的强状态空间。

第 10 章：统一流程与单线程协作

需求→Tasking→澄清→设计→计划→实现→Review→测试→验收→代码输出

Tasking 是前置知识整理

业务意图为什么要做，解决谁的问题。
用户路径用户如何触发、经过、完成。
验收标准什么证据说明真的完成。
边界条件哪些情况必须覆盖或明确排除。
依赖关系上游、下游、并行冲突。
风险未知项哪些要问，哪些要记录。

统一流程是资产生产线

输入输出状态角色检查点验收回写

流程中发现的新规则、新失败模式、新业务判断，要回写到 Skill、工程文件、Knowledge 或长期原则文件。

第 11 章：多线程协作流程

Epic→User Stories→依赖识别→Worktree 隔离→并行增量→验收汇合

第 12 章：自动化与无人值守

长时间、低人工干预；不是完全自主魔法。
任务边界Worktree自动测试浏览器验证可观测人工 Gate失败回流

长时间运行需要强状态

Feature List知道要完成哪些增量。
Progress File知道现在做到哪里。
Clean State知道从哪里安全恢复。

第 13 章：Agent 设计

角色协议上下文、权限、输入、输出。
契约完成条件、错误处理、副作用。
验收测试、trace、outcome、回写。

提速来自哪里

  • 上下文传递损耗下降。
  • 等待和协调成本下降。
  • 反馈更快，测试更靠前。
  • 失败更快变成 eval case。
  • 知识资产持续增长。

工具层改造

人类体验页面、说明、宽松输入、可读输出。
智能体体验JSON、schema、字段裁剪、稳定错误码、权限边界、输入防御。

最后带走一句话

AI-native workflow 是模型推理、流程约束和状态反馈的动态系统。
