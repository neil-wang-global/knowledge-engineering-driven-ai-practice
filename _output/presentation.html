第 0 章：为什么还要工程化与 SDD

模型更强一部分 planning、tool use、orchestration 被模型吸收。
问题迁移工程方法从能力补丁，迁移为治理工具。
现实风险模型太快开始写代码，需求、边界、验证还没想清。

来源：用户项目 framing；`references/web/24-mp-weixin-qq-com-s-b4Fh-3SQyWiB1HypUxk9wA.md`

AI-native workflow

模型推理生成方案、代码、解释、判断。
流程约束先想清楚，固定检查点，验收显化。
状态反馈可恢复、可检查、可交接、可回写。

第 1 章：软件基本功

SoCStory、ADR、Spec、Plan、State、Review、Eval 各司其职。
抽象 / OOPSkill、Rule、Capability、Agent Blueprint 是知识对象。
目录结构不是收纳方式，而是 Agent 可导航的知识地址空间。

上下文管理不是越多越好

给模型低噪声、可处理、可验证的上下文。

确定性的事交给确定性的工具；推理性的事交给模型或人。

第 2 章：MindFlow 的自我演进

Soul→Learning→Capability→Plan→Task State→Reflection

来源：MindFlow README-CN、SYSTEM、tasks、capabilities。

MindFlow 给我们的启发

  • 不要把所有知识揉成一个巨大 prompt。
  • 长期原则、项目知识、能力、状态、反思要分层。
  • Lazy loading 是流程设计，不只是省 token。
  • 任务状态让系统可恢复、可交接、可复盘。

软件系统的本质就是知识

传统软件代码、文档、测试、规则，是知识的固化。
Agent 系统上下文、状态、工具、反馈，也是知识的表达。

第 3 章：POMASA 与 MAS 的强状态

MAS 的关键不是“几个 Agent 聊天”，而是围绕状态推进的编排系统。

ProMaster 具体事实证据不足；本章事实主要使用 POMASA 资料。

POMASA 的工程信号

Blueprint / Pattern可声明、可复用的协作结构。
Filesystem Data Bus文件系统承载多角色产物流转。
Verifiable Data Lineage产物来源、验收、回流可追踪。

来源：POMASA README.zh-CN；POMASA SKILL。

第 4 章：状态控制三分法

强约束固定Rules、Hooks、必读文件、测试、提交检查。
建议式调度Skill、推荐流程、可选 checklist。
完全 AI 控制探索、低风险实验、一次性任务。

可靠性与灵活性的张力

强约束：可靠↔建议式：平衡↔AI 自主：灵活

这是工程经验分类；外部 workflows / agents 材料提供支撑。

第 5 章：Skill、工程文件与 Rules

Skill可复用能力；触发依赖模型判断，存在随机性。
工程文件结构化上下文；处在流程规定位置，更确定。
RulesHook 式强约束；稳定，但自由度低。

成熟 Skill 像函数

触发输入步骤输出权限副作用验证下游消费

来源：CLI for AI Agents；POMASA SKILL；用户项目 framing。

第 6 章：Knowledge 与 Story

Story 是动态知识，Knowledge 是静态 Story。

这是项目观点。外部材料用于支撑工程资产回写、Doc 与 Code 联动、eval 闭环。

User Journey 是知识固化过程

发现→组织→编码→验证→回写

第 7 章：Flyway 类比

Story = Migration带着时间性、变化意图和业务增量。
Knowledge = 当前状态所有已验证增量执行后的知识系统。

回写闭环

Story→代码文档测试SkillEval Case→Knowledge

第 8 章：验收 Skill 与 Story

Skill 验收触发、输入、步骤、输出、权限、副作用、验证、下游消费。
Story 验收业务意图、AC、依赖、边界、可测试性、知识沉淀。

不要只相信“我完成了”

第 9 章：角色而非人

BA业务意图与验收标准。
DEV设计、实现、工程风险。
QA / UI可测性、业务路径、用户体验。

Agent 承担检索、执行、验证、总结等动作。

单仓库原则

仓库是 BA / DEV / QA / UI / Agent 共享的强状态空间。

来源：用户项目 framing；OpenAI repository knowledge as system of record。

第 10 章：统一流程与单线程协作

需求→Tasking→澄清→设计→计划→实现→Review→测试→验收→代码输出

Tasking 是前置知识整理

  • 业务意图
  • 用户路径
  • 验收标准
  • 边界条件
  • 依赖关系
  • 风险与未知项

统一流程是资产生产线

输入输出状态角色检查点验收回写

Skill、工程文件、Knowledge、类似 `SOUL.md` 的长期原则文件，都随流程维护。

第 11 章：多线程协作流程

Epic→User Stories→依赖识别→Worktree 隔离→并行增量→验收汇合

多线程不是跳过流程

它是把统一流程复制到多个隔离工作流中，再通过依赖、状态和验收汇合。

第 12 章：自动化与无人值守

长时间、低人工干预；不是完全自主魔法。

长时间运行需要强状态

Feature List知道要完成哪些增量。
Progress File知道现在做到哪里。
Clean State知道从哪里安全恢复。

来源：Anthropic, Effective harnesses for long-running agents。

自动化的边界

任务边界Worktree自动测试浏览器验证可观测人工 Gate失败回流

第 13 章：Agent 设计

角色协议上下文、权限、输入、输出。
契约完成条件、错误处理、副作用。
验收测试、trace、outcome、回写。

提速来自哪里

  • 上下文传递损耗下降。
  • 等待和协调成本下降。
  • 反馈更快，测试更靠前。
  • 失败更快变成 eval case。
  • 知识资产持续增长。

展望

  • 框架稳定后，应沉淀为工具、模板、规则、测试、Skill 或 workflow。
  • 架构形态会被 AI 重新塑造。
  • AI 不是降低设计重要性，而是提高设计杠杆。
  • Agent Teams 是展望，不是成熟完全自治结论。
