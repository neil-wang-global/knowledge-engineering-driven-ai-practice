## 0. 这版怎么讲

这场分享不要讲成“AI 工具清单”，也不要讲成“概念百科”。它的叙事顺序应该是：

1. 先回到软件工程基本功：SoC、面向对象、目录结构。
2. 再说明这些基本功在 AI 体系下为什么变得更重要。
3. 然后讲 MindFlow、知识工程、POMASA、Skill、角色协作、敏捷、Agent 编排、提速和无人值守。

全场核心句：

> AI 编程不是推翻软件工程，而是把软件工程基本功放大了。结构越清晰，AI 越可靠；结构越混乱，AI 只是更快地制造混乱。

## 2. 完整讲稿正文

### 1. 开场：为什么 AI 编程要回到软件工程基本功

大家好，今天这个题目叫《从 AI Coding 到 AI 工作系统》。

标题听起来有点大，但我真正想讲的其实很简单：**AI 编程不是推翻软件工程，而是把软件工程基本功放大了。**

过去一年，很多团队都在试 AI Coding。大家一开始很兴奋，因为模型真的能写代码。你给它一个需求，它能扫仓库，能改文件，能补测试，甚至能一次生成一个小系统。

但过一段时间，大家又会遇到同一个问题：

> 为什么个人体感快了，团队交付没有等比例变快？

有人说是模型还不够强。有人说是 prompt 没写好。有人说是工具链还不成熟。

这些都对，但都不是最底层的问题。

最底层的问题是：**我们有没有把软件研发这件事，组织成 AI 能稳定参与的工作系统？**

AI 很像一个能力很强、记忆很短、执行很快、但非常依赖上下文的同事。你给它清晰的边界，它可以跑得很快；你给它混乱的上下文，它也会跑得很快，只是方向可能错了。

所以今天我不想一上来讲哪个工具好用，也不想先讲提示词技巧。我们先回到三个老东西：

- SoC，关注点分离；
- 面向对象，或者说抽象；
- 文件目录结构。

这些听起来都很传统。但我认为，AI 时代真正能把效率做出来的团队，最后拼的恰恰是这些基本功。

因为对 AI 来说，结构不是审美问题。结构决定它能不能看懂，能不能交接，能不能恢复，能不能验证，能不能长期运行。

### 2. SoC：从软件关注点分离到 AI 上下文分离

我们先讲 SoC，Separation of Concerns，关注点分离。

在传统软件工程里，SoC 是一个很朴素的思想：不要把所有事情搅在一起。业务逻辑归业务逻辑，基础设施归基础设施，展示层归展示层，测试归测试。

为什么要这么做？因为人脑处理复杂度是有限的。

如果一个订单系统里，支付逻辑、库存逻辑、页面展示、数据库访问、消息发送全部缠在一起，开发者改一个字段都要担心牵一发动全身。SoC 的价值就是把问题切开，让每一块有自己的职责。

到了 AI 时代，SoC 的价值没有消失，反而更重要。

因为 AI 也怕混乱。只是它不会像人一样说“我看不懂了”。它会继续回答，继续生成，继续改文件。问题是，它可能已经在错误的上下文里工作了。

所以在 AI 体系下，SoC 不只是代码设计原则，它变成了**上下文分离原则**。

什么意思？

假设我们让一个 Agent 修改用户注册功能。如果仓库结构清晰，它知道：

- 业务规则在 domain；
- 应用编排在 application；
- 数据库适配在 infrastructure；
- API 暴露在 controller 或 OHS；
- 测试在 test；
- 需求和验收标准在 docs 或 stories。

那它就能在正确范围里行动。

但如果这些东西混在一起，它就会开始猜：这个校验应该放 controller 吗？还是 service？这个字段是临时展示用，还是领域概念？这个测试失败是因为需求变了，还是代码错了？

人类资深工程师可以凭经验判断，AI 也能判断一部分，但判断成本会迅速上升。

所以，AI 时代的 SoC 可以这样理解：

> SoC 是给 AI 划出推理边界、修改边界和责任边界。

它告诉 AI：你现在扮演什么角色，应该读取哪些上下文，可以修改哪些文件，输出交给谁，完成后由谁验证。

这也是为什么后面我们讲多 Agent、分层 Plan、Skill、单仓库协作时，都会回到同一个原则：**先分清关注点，再让 AI 执行。**

没有 SoC，多 Agent 只是多个模型一起制造混乱。

### 3. 面向对象：从代码对象到 Markdown 对象

接下来讲抽象。更准确地说，我想讲的是面向对象。

很多人一听面向对象，会想到 class、interface、继承、多态。也有人会说，现在函数式、DDD、微服务、数据驱动这么多范式，为什么还要讲面向对象？

我这里讲的不是“必须用 Java class 写一切”。我讲的是面向对象背后的那个思想：

> 把一组稳定的职责、状态和行为，组织成一个可以被命名、被引用、被协作的对象。

对象的价值，不只是代码复用。对象最大的价值是**降低认知成本**。

举个例子。我们说 `Order`，不是因为它只是一个类名，而是因为它把一堆事情收束起来：订单有哪些状态，什么时候可以支付，什么时候可以取消，取消会影响哪些库存和退款规则。

一个好对象，是一个小的知识容器。

AI 时代，这个思想可以扩展到代码之外。

我们写 Markdown，也应该有面向对象意识。

比如一份需求文档，如果只是几页自然语言，AI 很难稳定消费。但如果它被组织成结构化对象，就不一样了：

```markdown
# UserStory: 企业用户注册

## Goal
让企业管理员可以创建企业账号，并邀请成员加入。

## Actors
- 企业管理员
- 普通成员
- 系统管理员

## Preconditions
- 管理员已完成手机号验证
- 企业名称未被占用

## Acceptance Criteria
- 当企业名称重复时，系统返回明确错误
- 当手机号未验证时，不允许创建企业
- 创建成功后生成默认团队空间

## Out of Scope
- 企业认证审核
- 付费套餐绑定
```

这份 Markdown 就不只是“文档”。它已经像一个对象了。

它有名字，有职责，有边界，有输入，有输出，有不变量，有验收条件。Agent 读取它时，不是在读一段散文，而是在读取一个可消费的工程对象。

再比如 Skill 也是对象。

一个 Skill 应该说明：什么时候触发，需要什么输入，按什么步骤执行，输出写到哪里，失败怎么办，如何验证。

```markdown
# Skill: code-review

## Trigger
当一个功能实现完成、准备合并前使用。

## Inputs
- PR diff
- 需求说明
- 测试结果

## Workflow
1. 检查需求覆盖
2. 检查代码边界
3. 检查测试缺口
4. 输出阻塞问题和建议问题

## Output
写入 review-report.md
```

这同样是面向对象，只不过对象不再是一个 class，而是一个可执行的知识单元。

所以我想强调一个观点：

> AI 时代的面向对象，不只发生在代码里，也发生在 Markdown、目录、文档、Skill、任务状态和 Agent 设计里。

这和软件范式并不冲突。函数式强调纯函数和数据流，DDD 强调领域语言，事件驱动强调状态变化，微服务强调边界和契约。这些范式在 AI 体系下仍然非常有价值，因为它们都在做同一件事：**保持结构稳健，让变化可控，让上下文可引用。**

如果我们把 Markdown 写成一堆流水账，AI 就只能猜。如果我们把 Markdown 写成对象，AI 就能引用、检查、传递和复用。

这就自然引出下一件事：目录结构。

### 4. 目录结构：AI 的认知地图

目录结构以前常被当成项目收纳方式。src 放代码，test 放测试，docs 放文档。

但在 AI 工作系统里，目录结构的意义要更大。

它是 AI 的认知地图。

因为 Agent 不像团队里的老员工，它不知道“那个文档一般在飞书里”“那个设计稿在某个群公告里”“那个历史决策在某个同事脑子里”。它能稳定读取的，是路径、文件、状态和结构。

所以，目录结构应该告诉它：

- 需求在哪里；
- 计划在哪里；
- 当前状态在哪里；
- 中间产物在哪里；
- 测试结果在哪里；
- 评审记录在哪里；
- 最终输出在哪里；
- 哪些知识已经固化，哪些还在草稿。

MindFlow 就是一个很好的例子。它不是只说“让 AI 记住任务状态”，而是要求任务有正式目录，有 plan，有 step，有 state，有 cache，有 output，有 reflection。状态不是靠模型记忆，而是落在文件里。

POMASA 也是类似思路。它有 agents，有 references，有 workspace，有 _output。多智能体不是靠互相转述维持秩序，而是靠目录和文件维持秩序。

这就是强状态。

弱状态是什么？聊天上下文、口头约定、临时 prompt、模型说“我记住了”。

强状态是什么？文件、目录、计划、任务状态、测试结果、评审记录、引用来源、输出产物。

弱状态适合探索，强状态适合交付。

AI 编程一旦从 demo 进入真实工程，就必须从弱状态走向强状态。

### 5. 上下文管理：尽在上下文

现在我们可以讲上下文管理了。

我觉得 AI 辅助编程最核心的一句话是：**尽在上下文。**

JoyCode 的企业级 AI Coding 实践也把“尽在上下文”作为核心命题。它面对的不是一个玩具项目，而是复杂业务场景、丰富用户画像、复杂长任务和超大代码仓。对这种企业级场景来说，问题不是“模型会不会写代码”，而是它能不能在正确上下文里理解代码。

这个点非常关键。

我们平时说“模型很强”，其实容易忽略一个前提：模型只有在看见正确材料时才强。

如果上下文缺了，它会补。补得有时候还挺像真的。

如果上下文错了，它会沿着错误上下文认真推理。

如果上下文太多，它会被噪声稀释，关键约束可能埋在中间，被忽略。

如果上下文彼此冲突，它会选择一个看起来顺的解释，而不一定是业务上正确的解释。

这就是为什么企业级 AI Coding 不能只靠“把仓库丢给模型”。

一个真实企业仓库里有什么？

- 历史代码；
- 旧架构和新架构；
- 临时补丁；
- 已废弃但还没删的接口；
- 业务特例；
- 自研框架；
- 约定俗成的命名；
- 没写进文档的规则；
- 多个团队留下的风格差异。

这些东西都在仓库里，但不是所有东西都应该在同一时刻进入模型上下文。

所以，上下文管理不是越多越好，而是要回答五个问题：

1. 什么信息应该出现？
2. 什么时候出现？
3. 出现给哪个角色？
4. 以什么格式出现？
5. 如何验证它被正确消费？

SoC 是减少不相关上下文。
面向对象是压缩上下文。
目录结构是索引上下文。
Skill 是按需加载上下文。
Agent 是隔离上下文。
Script 是把不需要推理的事情移出上下文。
测试和评估是检查上下文消费结果。

这里有个很实用的原则：**知识不是一次性投喂，而是渐进加载。**

不要一上来把所有规范、所有文档、所有历史都塞给模型。那不是知识工程，那是上下文污染。

好的做法是像程序运行一样：进入哪个模块，加载哪个模块的规则；执行哪个角色，读取哪个角色需要的上下文；到了验收阶段，再加载验收标准和测试结果。

MindFlow 的 lazy loading 就是这个思想。它要求模块规则在进入模块时读取，而不是任务一开始全部预加载。

这其实很像我们写程序时的依赖管理。你不会让一个函数 import 全世界，你只会让它依赖它真正需要的东西。

AI 工作流也一样。

### 6. MindFlow：把认知过程工程化

讲完这些基本功，我们再看 MindFlow 就更容易理解了。

如果代码系统需要 SoC、面向对象和目录结构，那么 AI 工作系统也需要认知层面的 SoC、对象和目录结构。

MindFlow 可以理解为：把一个 AI 做事、学习、反省、固化能力的过程，工程化成一套结构。

它里面有几个核心对象。

第一个是 Soul。

Soul 不是玄学。你可以把它理解成系统的最高约束：什么是好结果，什么方法被偏好，什么不能做，什么值得学习。

人类团队也有 Soul，只是以前它可能藏在资深工程师的经验里，藏在 code review 的口头反馈里，藏在“我们团队一般不这么写”的默契里。

AI 要稳定工作，这些默契就不能只待在人脑里。它们要结构化。

第二个是 Learning。

Learning 不是存资料。很多团队一说知识库，就开始堆文档。但 MindFlow 的思路不是“堆资料”，而是让知识经过读取、反省、审核、固化，最后变成可调用能力。

第三个是 Capability。

Capability 是能力对象。不是一段散文，而是一个可以在任务中被调用的能力单元。

第四个是 Plan / Step。

任务不能只靠一句“帮我做完”。它要拆成计划，计划要拆成步骤，步骤要有输入、输出、状态和能力要求。

第五个是 State。

State 非常重要。AI 长任务最怕什么？怕跑到一半不知道自己在哪，怕失败后恢复不了，怕换一个会话就丢上下文。

State 的价值就是让任务可恢复、可审计、可交接。

所以 MindFlow 不是普通知识库，也不是普通多 Agent 编排器。它真正做的是：**把认知过程工程化。**

这也是我想给 AI-native workflow 的定义：

> AI-native workflow = 模型推理 + 流程约束 + 状态反馈的动态系统。

模型是动的流程，流程是动的模型。

模型会根据上下文和状态不断推理；流程会通过约束和反馈不断塑造模型下一步怎么行动。

### 7. 知识工程：不是提示词工程

这里可以接上熊节那篇《不是提示词工程，是知识工程》。

那篇文章讲了五个支柱：问题导向、价值框架、信息储备、理论方法、表现形式。

这个说法很好，因为它把“怎么问 AI”往前推了一层。

很多人以为提示词工程是在研究怎么把一句话写漂亮。但真正重要的问题是：

- 你到底要解决什么问题？
- 你用什么价值标准判断结果好坏？
- 你准备了哪些信息？
- 你用什么方法分析？
- 你最终要以什么形式交付给谁？

这些问题没想清楚，prompt 写得再长也不稳。

这和 MindFlow 可以对齐：问题导向对应任务识别，价值框架对应 Soul，信息储备对应 Learning，理论方法对应 Capability，表现形式对应 Output。

我还想加一个更工程化的说法：

> 知识是 story diff，user journey 是知识。

这句话可能一开始听起来有点抽象，我解释一下。

一个用户故事刚写出来时，通常是不完整的。BA 和客户聊了一轮，补了一个验收条件，这是 diff。DEV 实现时发现一个边界条件，这是 diff。QA 测试时发现一个异常路径，这是 diff。UI 走查时发现用户路径不顺，这是 diff。客户验收时说“这个场景我们其实还要支持”，这也是 diff。

这些 diff 才是真正有价值的知识。

因为它们不是静态资料，而是团队在真实工作中，对业务、系统和用户理解发生变化的证据。

如果这些 diff 只留在聊天记录里，下次还会丢。真正的知识工程，是把这些 diff 沉淀成：

- 工程文档；
- Skill；
- Capability；
- 测试用例；
- eval case；
- checklist；
- workflow pattern。

这就是 AI 时代知识沉淀的核心。

### 8. POMASA：MAS 工程样本

接下来讲 POMASA。

POMASA 可以作为一个多智能体工程样本来看。

很多人理解多 Agent，会想象成“我开几个模型，让它们互相讨论”。但真正可交付的 MAS 不能只是群聊。

POMASA 的思路是：先有 user_input，再生成 Agent Blueprints，再有 references 和 methodology，再通过 orchestrator 组织工作流，最后把研究过程和输出落到 workspace 和 _output。

这几个东西很重要：

- Blueprint 定义角色和行为；
- references 提供知识来源；
- methodology 提供方法论；
- workspace 保存中间产物；
- _output 保存最终交付；
- orchestrator 负责阶段推进。

这就说明，多智能体系统的可靠性不来自 Agent 数量，而来自结构。

如果没有文件、目录、状态、角色契约和数据血缘，多个 Agent 只是多个不稳定的对话。

POMASA 还有一个特别重要的原则：调用子 Agent 时，不要由上游转述它的完整职责，而是让子 Agent 自己读取完整 Blueprint。

这听起来像细节，其实很关键。

因为长链路里最容易出问题的是指令失真。你转述一次，丢一点；再转述一次，又丢一点。最后子 Agent 执行的已经不是原始任务，而是被压缩、改写、误解后的任务。

所以，MAS 工程的一个底层原则是：**关键约束必须有权威来源，不能靠口口相传。**

### 9. Skill / 工程文档：本质是可执行知识

现在讲 Skill 和工程文档。

以前工程文档经常有个尴尬处境：写的时候很认真，用的时候没人看。或者新人看了，但不知道哪些是必须遵守，哪些只是背景说明。

AI 加入以后，文档有了新的价值。

文档不再只是“给人看的说明”，它可以变成 Agent 执行时读取的规则、模板、检查清单和验收标准。

但前提是：文档要写得像可执行对象。

一个好 Skill 不应该只是“请你认真做代码评审”。它应该说清楚：

- 什么时候触发；
- 输入是什么；
- 要读哪些文件；
- 执行哪些步骤；
- 输出到哪里；
- 什么情况算完成；
- 什么情况必须停止；
- 如何验证。

这和工程文档是同一件事。

好的工程文档也应该有对象感。比如 API 文档不只是列接口，而要说明调用方、前置条件、状态变化、错误路径和兼容性。测试规范不只是说“要写测试”，而要说明什么场景必须测，谁负责测，失败如何回流。

所以 Skill 和工程文档的共同本质是：**把人的经验和判断标准变成可执行知识。**

这也是为什么我们前面讲 Markdown 要面向对象。

如果 Markdown 没有结构，它只是文字。
如果 Markdown 有职责、输入、输出、边界、验收，它就是 AI 工作系统里的对象。

### 10. 角色、敏捷与单仓库协作

接下来讲团队协作。

AI 工作流里，不要一上来讨论“AI 会不会替代某个人”。更准确的讨论方式是：角色会怎么变化。

BA 是角色，DEV 是角色，QA 是角色，UI 是角色。一个人可以承担多个角色，一个 Agent 也可以承担某些角色活动。但角色边界不能消失。

这时候敏捷反而更重要。

很多人会觉得，AI 这么快了，是不是可以绕过敏捷？我认为不是。AI 不应该绕过敏捷，而应该把敏捷工件变成可执行、可检查、可回流的强状态。

Story 是什么？不是一段需求描述，而是业务意图的最小可交付表达。

Acceptance Criteria 是什么？不是附属说明，而是 QA 生成测试、DEV 设计边界、Agent 理解目标的共同输入。

DoR，Definition of Ready，可以变成 Agent 开始执行前的上下文完备性检查：

- 需求是否清楚？
- 验收标准是否可测试？
- 依赖是否明确？
- 相关代码边界是否找到？
- 设计约束是否存在？
- 风险是否记录？

DoD，Definition of Done，可以变成 Agent 完成后的验收 checklist：

- 代码是否实现？
- 测试是否通过？
- 文档是否更新？
- review note 是否处理？
- eval case 是否补充？
- 发布说明是否生成？

Sprint 也不只是时间盒。它是人和 Agent 协作的节奏控制器。

计划阶段明确任务边界和依赖。执行阶段通过 task state、checkpoint 和 PR 管理进展。Review 阶段检查 outcome，而不是只看模型说“完成了”。Retro 阶段把失败、返工和遗漏沉淀为 Skill、测试和 eval case。

这就引出单仓库原则。

单仓库不是说所有团队必须用一个巨大仓库。这里说的是：关键交付状态要处在一个可追溯、可读取、可检查的共享空间里。

这个空间里不应该只有 src 和 test，还应该有：

- PRD / user story；
- UX / UI spec；
- domain model；
- plan；
- task state；
- test cases；
- review notes；
- eval cases；
- release notes；
- knowledge updates。

这样 BA、DEV、QA、UI 和 Agent 才能围绕同一个事实空间协作。

BA 检查 DEV 是否实现业务意图。DEV 检查 BA 的需求是否可实现。QA 检查验收标准是否可测试。UI 检查用户路径是否合理。Agent 在边界内执行、检查、回填。

这不是让流程变重，而是让协作变得可执行。

### 11. Agent 设计与提速

有了角色，我们再讲 Agent 设计。

Agent 不是 prompt 名字。不是我写一个“你是资深工程师”，它就变成了可靠工程师。

Agent 是角色、权限、上下文、输入、输出、状态、验收和交接的组合。

设计 Agent 要问：

- 它扮演什么角色？
- 它能读什么？
- 它能改什么？
- 它不能做什么？
- 输入从哪里来？
- 输出写到哪里？
- 谁消费它的输出？
- 谁验收它？
- 它遇到不确定性时怎么办？

这里可以区分几类 Agent：

- Thinker Agent：只做设计，不改代码；
- Worker Agent：按设计实现，不随便改设计；
- Reviewer Agent：独立检查，不共享实现上下文；
- Orchestrator：编排流程，不混入具体实现细节。

这种拆分的目的不是形式好看，而是为了保持上下文干净。

同一个 Agent 如果又设计、又实现、又评审、又修复，它的上下文会越来越脏。最后它不是更智能，而是更容易自我合理化。

提速也是同一个道理。

AI 编程提速，不是让模型疯狂输出更多代码。真正的提速是控制代码产出水平：任务粒度可控，上下文可控，边界可控，输出可控，验证可控。

如果设计错了，AI 只会更快地生成错误实现。AI 不是降低设计重要性，而是提高设计杠杆。

这句话一定要讲清楚。

以前一个错误设计，可能只影响几百行代码。现在一个错误设计，Agent 可能十分钟生成几千行错误实现。所以 AI 越强，设计越重要。

### 12. 测试、评估与检测

接下来讲测试。

没有测试的提速，是把风险从开发阶段推迟到验收阶段。

在 AI 时代，测试不只是质量保障。测试是 AI 的反馈环境。

模型看不见“业务上对不对”，除非我们把对错变成可检测信号。

能确定的事情，就不要交给模型猜。

状态管理、契约校验、进度查询、lint、test、文件结构检查、frontmatter 校验、输出格式校验，这些都应该交给确定性工具。

有一句原则很好用：

> Skill 编排流程，Agent 处理不确定性，Script 处理确定性。

模型最擅长处理开放问题。脚本最擅长处理确定规则。不要让模型去“创造性解释”一个 failed 状态是不是“基本完成”。

反思也一样。

“我检查过了，没问题”不算反思。

真正的反思必须带证据：引用哪个目标、哪个章节、哪个契约、哪个文件、哪个测试、哪个输出。

失败也不能只当事故。失败是 eval case 的来源。

客户反馈、QA bug、review 问题、Agent 跑偏、上下文缺失，都应该回流成测试、checklist、skill rule 或 eval case。

Agent eval 里有几个概念可以轻轻带一下：Task 是测试任务，Trial 是一次尝试，Grader 是评分器，Transcript 是运行轨迹，Outcome 是最终状态。

这里重点不是术语，而是思路：

> 不要只评估 Agent 说了什么，要评估环境最后变成了什么。

企业交付关心的不是偶尔成功，而是稳定不坏。pass@k 可以说明“多试几次能不能成功”，pass^k 更关心“连续多次是否都成功”。真实生产更接近后者。

### 13. AI 编排、工具层和无人值守

最后把编排方式放在一起看。

纯 Vibe Coding 适合小任务，快，反馈直接。但它的问题是上下文不可控，结果难复用。

轻量 SDD 多了 brainstorming、plan、task、impl，适合千行级任务。有 plan with file，也能断点续跑。

但企业级任务会遇到集中 plan 的瓶颈。所有架构、框架、依赖、接口、领域边界都塞进一个 plan，模型会被撑爆。

这时要进入 Sub-agent 和分层 Plan。

Sub-agent 解决执行上下文污染。分层 Plan 解决规划上下文爆炸。

比如按 OHS、Application、Domain、Infrastructure 分层，每层只关心自己的职责。上游导出契约，下游依赖契约。下游依赖必须是上游导出的子集。

这就是契约驱动的跨层一致性。

工具层也要改造。

人类 CLI 不等于 Agent CLI。人类体验追求好探索、好容错；智能体体验追求可预测、可约束、可防御。

Agent 是不受信任的操作员。它能力很强，但会幻觉，会误读，也可能被提示注入影响。

所以面向 Agent 的 CLI 应该支持：

- JSON payload；
- schema 自省；
- field mask；
- pagination；
- NDJSON；
- --dry-run；
- --sanitize；
- 结构化错误；
- 明确权限边界。

这样才有可能进入长时间自主运行。

但无人值守不是放飞 AI。无人值守是把人的责任前置到 workflow、权限、状态、测试和评审设计里。

越想自动化，越要有强状态、checkpoint、自动测试、独立 review、人类关键决策点、可观测和回滚机制。

### 14. 展望：Agent Teams、架构变化和框架的终点

最后讲三个展望。

第一，Agent Teams。

现在很多多 Agent 系统还是主从式编排：主 Agent 拆任务，子 Agent 执行。未来可能会走向协作式 Agent Team。Worker 发现设计有问题，可以向 Thinker 开卡；Reviewer 可以触发返工；QA 可以阻塞发布。角色之间通过强状态和契约双向协作。

第二，架构形态会被 AI 重塑。

今天很多微服务、领域划分、服务拆分，是为人的认知上限设计的。未来架构决策可能会从“人能理解多少”，变成“AI 在什么粒度上生成和维护代码最高效”。当然，性能、安全、部署、组织边界仍然重要，但认知约束会变化。

第三，框架是为了消灭自己而存在。

分层 Plan、契约校验、渐进加载、权限分级，都是当前模型能力不足时的工程补丁。未来模型更强，这些框架形态会变化。

但搭建这些框架的过程不会白费。它会迫使团队想清楚：哪些事需要推理，哪些事需要规则，哪些信息应该何时出现，哪些结果必须被验证。

框架会过时，工程判断不会过时。

所以回到开头：AI 编程的终局，不是人人都会写 prompt，而是团队拥有持续把经验、流程、工具和反馈转化为 AI 可执行能力的系统。

## 5. 附录 / Q&A 备选材料

### Q1：为什么不在正文里放很多 [SRC-xxx]？

现场演讲不适合把引用编号塞进正文，会打断听众理解。正文中保留来源归属，完整引用放在最后。正式发布版可以再加脚注。

### Q2：Promptfoo、Braintrust、LangSmith、Langfuse 怎么定位？

它们属于 eval / observability / trace / experiment management 层。主讲不展开产品比较，只强调评估系统要记录 Task、Trial、Trace、Outcome，并能重复运行。

### Q3：SWE-bench、Terminal-Bench 该怎么讲？

它们可以作为 benchmark 例子，但不要把 benchmark 等同于生产可用性。Benchmark 证明“能做”，生产系统还要证明“稳定做、可回滚、可观测、可验收”。

### Q4：上下文压缩有哪些技术？

可以讲 summary、tool output pruning、sliding window、pinning、field mask、分页、NDJSON。重点不是技巧，而是区分哪些信息必须完整保留，哪些可以压缩，哪些应该由工具确定性查询。

### Q5：什么时候用 Sub-agent？

当任务边界清晰、输入输出可文件化、执行结果可独立验收时适合用 Sub-agent。边界不清时，先做 plan 和 contract，不要急着并行。

### Q6：AI 会不会替代 BA / DEV / QA / UI？

更准确的说法是角色会被重组。Agent 可以承担部分角色活动，但角色边界、验收责任和业务判断不会消失。

### Q7：无人值守什么时候安全？

当任务边界明确、权限可控、状态可恢复、测试可自动运行、关键决策有人类 gate、失败可回滚时，可以逐步增加无人值守程度。
