# 知识工程驱动的 AI 实践

项目：ai-work-system-speech-20260518

> 这是一份 60-90 分钟中文演讲稿。主线只有一条：如何构建统一流程，以及如何构建一个高质量的统一流程。

## 开场

各位好，今天分享的题目是《知识工程驱动的 AI 实践》。这个题目听起来有点大，但我想讲的不是一个抽象概念，也不是一组工具清单。我想讨论一个很具体的问题：当 AI 已经能写代码、能读仓库、能调用工具、能做一部分规划时，我们怎样把它放进一个稳定、可协作、可演进的工作系统里。

很多团队第一次接触 AI Coding，会从提示词、模型选择、编辑器插件开始。这些当然有价值，但如果只停在这里，很快会遇到同一个问题：模型确实能产出代码，而且产出得很快；可团队不一定知道它为什么这么写，边界有没有覆盖，需求有没有被误解，测试有没有验证到关键路径，下一次换一个 Agent 或换一个人接手时，状态还能不能恢复。

所以今天我不会把重点放在“如何让模型多写一点代码”。我更关心的是：怎样让模型的推理结果可以被流程约束，被状态记录，被测试验证，被团队交接，并且最终沉淀成工程资产。用一句话概括，AI-native workflow 是一个动态系统：模型负责推理，流程负责约束，状态负责让推理结果可以被恢复、检查和交接。这个定义是本次分享的项目 framing，不是行业标准术语，但它会贯穿今天所有章节。

## 第 0 章：为什么还要工程化与 SDD

我们先回答一个最容易被问到的问题：模型越来越强，为什么还要工程化？为什么还要 SDD、流程、规则、状态、验收？

这个问题不能只从“模型会不会写代码”来回答。今天很多团队的真实体感，已经不是 AI 不会写，而是 AI 太快开始写。需求还没澄清完，它已经开始实现；边界条件还没想清楚，它先自己补默认值；方案只有一个模糊轮廓，它已经把执行往前推。表面看很积极，实际上是在用流畅感掩盖过早收束。

所谓过早收束，不是模型输出慢，也不是模型语气不自信。恰恰相反，它常常表现得很自信。它会很快选定一条路径，很快补齐缺失信息，很快生成代码，很快宣布完成。但它没有充分展开问题空间：异常路径有没有覆盖，权限边界有没有想清楚，状态一致性有没有保证，验收条件有没有被定义，测试有没有验证到真正的业务路径。这种问题在演示场景里不一定明显，在真实工程里会变成返工、线上风险和知识漂移。

顶尖模型确实在变化。它们越来越能自己拆任务、调工具、维持更长轨迹，也能把一部分 planning、tool use、orchestration 吃进模型内部。以前很多外置提示词、外置流程、外置角色，是在给模型补能力；现在最强的一档模型，已经把其中一部分能力内化了。所以我们不能把 Prompt Engineering、SDD、MAS 当成永远固定的一套仪式。

但这不等于方法论失效。更准确地说，方法论的必要性开始分层。你用什么模型，花多少钱，任务有多开放，错误代价有多高，是否需要协作、审计、复用和治理，决定了工程结构要多强。一次性的个人探索，强模型直接交互可能已经够用；多人协作、预算受限、需求含糊、边界复杂、结果要追责的任务，就必须把外部结构建起来。

尤其是中端或低端模型、上下文不足、反馈不足、预算受限的场景，过早收束会更早浮出来。它们更依赖清晰的任务边界，更依赖明确输入，更依赖检查点和验收标准。你不给结构，它就会自己补结构；你不给边界，它就会自己猜边界；你不给验证，它就会用“看起来合理”的输出来代替验证。这不是态度问题，是反馈不足和约束不足的问题。

这就是 SDD 的价值。SDD 不是神秘地提高模型智力，它是把“想清楚”和“开始干”分开。它要求我们先把 what、why、guardrails、验收标准和关键边界显化，再进入 how。它对抗的不是模型愚笨，而是项目里的模糊、短视和过早执行。

Prompt Engineering 的价值也在迁移。它越来越不像“教模型怎么想”的咒语，而更像任务接口设计：目标是什么，输入范围是什么，禁止事项是什么，成功标准是什么，输出结构是什么。提示词不是魔法，它是任务契约的一部分。

MAS 的价值也不能理解成“多叫几个模型一起想”。真正有工程价值的 MAS，是把复杂任务拆成角色、上下文、权限、阶段、产物和验收。它让系统知道谁负责澄清，谁负责实现，谁负责核验，谁能读什么，谁能改什么，中间产物放在哪里，失败如何回流。它解决的是上下文隔离、权限边界、并行推进和审计问题。

Rules 和 Hooks 则解决另一类问题：哪些动作不能交给模型临场决定。比如提交前必须跑检查，引用必须可追溯，敏感操作必须确认，测试失败不能宣称完成。这些动作要有 deterministic control。模型可以推理，但固定边界要由流程和工具兜住。

所以，AI 编程不是推翻软件工程，而是放大软件工程基本功。AI 越快，越需要流程控制它什么时候开始；AI 越能做，越需要状态记录它做到了哪里；AI 越像执行者，越需要权限边界和验收边界。

今天的主题因此不是“AI Coding 技巧”，而是如何构建统一流程，如何构建一个高质量的统一流程。统一流程从输入到输出：需求进来，经过 Tasking、澄清、设计、计划、实现、Review、测试和验收，最后代码出去，并且验收测试通过。所有后面的概念，SoC、MindFlow、POMASA、Skill、Rules、Knowledge、Story、Worktree、Eval、Harness，都会回到这条流程上。

## 第 1 章：软件基本功：SoC、抽象/面向对象、文件目录结构

既然工程结构仍然重要，我们就先回到几个很老的基本功：SoC、抽象/面向对象、文件目录结构。

我知道这些词听起来太基础了。很多技术团队听到 SoC，会觉得这是入门课；听到面向对象，会想到类、接口、继承；听到目录结构，会觉得只是项目怎么摆文件。但在 AI 编程里，这些基本功会被重新放大。因为 Agent 不是凭空工作的，它通过上下文理解系统，通过文件找到知识，通过命名判断边界，通过目录推断责任。你给它一个混乱的知识空间，它就会在混乱里推理；你给它一个边界清楚、位置稳定、引用明确的空间，它才有可能稳定地工作。

SoC 在 AI 文件工程里，首先不是代码分层，而是知识分工。一个 Story 应该描述一次业务增量，说明用户要什么、为什么要、验收标准是什么。一个 ADR 应该记录架构决策，说明为什么选择 A 不选择 B。一个 Spec 应该表达需求契约，说明输入、输出、边界和不可做的事情。一个 Plan 应该拆执行路径，说明先做什么、后做什么、每一步完成条件是什么。一个 State 应该记录当前进度，告诉下一个人或下一个 Agent 现在做到哪里。一个 Review 应该记录审查意见，一个 Eval 应该记录行为验证。

这些文件如果职责不清，就会发生一件很麻烦的事：所有文档都像权威，但没有一个文档真正权威。Story 里写了设计，设计文档里又复制了需求，Plan 里又补了一段验收标准，Review 里又改了一条业务规则。人还能靠记忆和会议补救，Agent 不行。Agent 会很认真地读取每一份材料，然后在互相矛盾的事实里做一个看起来合理的综合。这个综合越流畅，风险越隐蔽。

所以，AI 文件工程的第一条原则是：每个知识对象只负责自己的范围，小而精，低冗余。依赖关系靠 ID、链接和引用来维护，不靠复制粘贴来维持上下文。复制看起来方便，其实是在制造事实分叉。一旦事实分叉，Agent 就不知道哪个版本是当前状态，团队也不知道该改哪一份。

抽象和面向对象也可以迁移过来。我们过去讲对象，会说一个对象应该有清晰职责、稳定接口、内部实现可以变化、外部依赖不应该知道太多细节。Skill、Rule、Capability、Agent Blueprint、工程约定、验收模板，本质上也应该这样设计。一个 Skill 不能只是“这里有一段经验，你看着办”。它应该像一个有接口的对象：什么时候触发，输入是什么，输出是什么，副作用是什么，权限边界在哪里，下游怎么消费，失败时怎么办。

如果一个知识对象没有接口，它短期能用，长期不可维护。今天你知道这段提示词是干什么的，明天换一个人就不知道；今天这个 Agent 触发了它，明天另一个 Agent 没触发；今天输出格式下游能吃，明天多了两句话下游就坏了。很多 AI 工作流的不稳定，不是模型突然变差，而是知识对象没有被工程化。

目录结构也不是收纳方式。它是知识对象的地址空间。对人来说，目录结构影响查找效率；对 Agent 来说，目录结构影响理解路径。一个仓库里如果需求、设计、状态、测试、脚本、引用资料都混在一起，Agent 必须靠搜索和猜测建立地图。相反，如果目录本身表达了职责：`stories/` 放业务增量，`adr/` 放决策，`plans/` 放执行路径，`state/` 放当前状态，`evals/` 放验证用例，`skills/` 放可复用能力，那么 Agent 进入仓库时就不是面对一堆文件，而是进入一个有地址、有边界、有导航的知识空间。

这里可以想象一个嵌套知识对象。最外层是仓库，它是团队共享的强状态空间。仓库下面有长期原则，有当前项目，有用户故事，有架构决策，有执行计划，有测试和评估。某个 Story 里只保留业务意图和验收标准；它引用某个 ADR 来说明架构约束；它引用某个测试或 eval case 来说明验收方式；它引用某个 Skill 来说明执行时必须遵守的步骤。这样，Agent 不是把所有材料读成一团，而是在一个图结构里沿着引用加载必要知识。

这就是渐进加载的基础。上下文管理不是把所有东西一次塞给模型。很多时候，上下文太多反而会降低质量，因为噪声变多、权威关系变模糊、当前任务边界被冲淡。更好的方式是让入口文件像地图：告诉 Agent 去哪里找任务、去哪里找规则、去哪里找测试命令、去哪里找当前状态。真正需要时再加载细节。

这里可以再讲得具体一点。一个好的 AI 工程仓库，不应该让 Agent 先读二十份文档再猜重点，而应该让它先读一个入口：当前任务是什么，必须遵守哪些规则，相关材料在哪里，当前状态是什么，验证命令是什么。如果任务需要业务背景，就沿着 Story 去读；如果需要架构约束，就沿着 ADR 去读；如果需要执行步骤，就沿着 Plan 去读；如果需要验收，就沿着测试和 eval 去读。这样上下文是按需展开的，不是一次性倾倒的。

这和人类 onboarding 很像。你不会把公司所有文档一次性发给新人，然后要求他立刻理解项目。你会给他地图、入口、当前任务、关键约束和联系人。Agent 也一样。区别只是 Agent 的上下文窗口更脆弱，它更容易被无关材料带偏。所以我们设计文件和目录时，其实是在设计 Agent 的认知路径。

所以，软件基本功在 AI 时代没有过时。它们变得更重要。AI 编程不是降低设计重要性，而是提高设计杠杆。过去一个糟糕目录影响的是人；现在它影响的是人和 Agent。过去一个职责混乱的文档只是难维护；现在它会直接污染模型上下文。过去一个边界不清的接口会让开发者困惑；现在它会让 Agent 生成错误实现。

这一章的结论很简单：AI 文件工程首先是工程。边界清楚、对象清楚、位置清楚，系统才容易维护，也才容易被 Agent 操作。后面讲 MindFlow、POMASA、Skill、Rules、Knowledge，本质上都在讲如何把这些基本功扩展到 AI 工作系统里。

## 第 2 章：MindFlow 的自我演进与知识工程

有了文件工程的基本功，我们再看 MindFlow。这里我不把 MindFlow 讲成项目说明书，也不逐项介绍它有哪些目录。我更想讲它背后的设计思想：一个 AI 系统如果要长期工作，就不能只依赖一次对话。它需要长期原则，需要学习资料，需要能力单元，需要当前计划，需要任务状态，还需要复盘和回写。

很多人做 AI 工作流时，第一反应是写一个很长的 prompt。把背景、规则、经验、需求、步骤、注意事项全部塞进去，希望模型一次读完以后就稳定执行。这个做法在小任务里能跑，在复杂任务里很快会遇到问题。上下文越来越大，模型越来越难判断什么是长期原则、什么是当前任务、什么只是历史残留。最后 prompt 变成一个知识垃圾场，看起来信息很全，实际权威关系不清。

MindFlow 的启发在于，它把知识拆成不同层次。Soul 承载长期原则和方向，Learning 承载学习材料和外部知识，Capability 承载可复用能力，Plan 承载当前任务的执行路径，Task State 承载当前进度，Reflection 承载复盘结果。每一层都有自己的职责，不靠重复材料维持上下文。

这其实就是 SoC 的延伸。长期原则不应该散落在每个任务里，能力单元不应该复制在每个计划里，当前状态不应该只留在聊天窗口里，复盘结果不应该只停留在一句“下次注意”。如果这些东西不分层，系统就很难演进。每次任务都像从零开始，每次失败都只是一次失败，每次经验都只存在于临时上下文。

MindFlow 还体现了一个非常重要的思想：文件不是简单的 input，最终文档也不是简单的 output。文件是知识的载体，目录、引用、状态、能力和复盘共同构成一个可以被 Agent 操作的知识空间。Agent 执行任务时，不只是读材料然后写结果。它是在这个知识空间里选择知识、组合知识、验证知识，并在完成后更新知识。

这也是为什么我说 MindFlow 不是一个文件夹设计，而是一个知识工程设计。它回答的是：AI 应该如何管理知识，如何加载知识，如何知道当前状态，如何把经验沉淀下来。这个问题比“提示词怎么写”更底层。提示词解决的是一次交互，知识工程解决的是系统如何持续变好。

如果只看一次任务，写一个好 prompt 可能就够了。但如果看十次、一百次任务，真正重要的是经验如何积累。今天某个 Agent 发现一个测试命令，明天是否还需要重新问？今天某个需求暴露一个边界条件，下一次 Story 是否能直接复用？今天某次失败说明某个 Rule 太宽或太窄，流程是否会调整？如果答案都是不能，那系统没有学习，只是在重复对话。

MindFlow 这类设计的意义，就是把一次次执行变成知识增量。它让任务状态不丢，让能力可以复用，让反思可以回写，让系统从“每次重新开始”变成“沿着已有知识继续演进”。这就是自我演进的工程含义。它不是说系统神秘地拥有自我意识，而是说系统有机制把经验变成下一次执行的上下文资产。

从这里可以引出一个更大的判断：软件系统的本质就是知识，智能 AI Agent 系统也是知识。代码是知识，因为它固化了业务规则和技术决策；文档是知识，因为它解释了意图和约束；测试是知识，因为它描述了系统应该如何行为；规则是知识，因为它把团队经验变成强约束；状态是知识，因为它告诉系统当前在哪里；运行反馈也是知识，因为它暴露真实世界里的成功和失败。

所谓“知识工程”，并非什么高深莫测的理论，它更像是一种实践方法论，强调在与AI协作的过程中，围绕以下五个关键环节进行有意识地规划、构建与传递。这些环节的工作，恰恰是AI本身无法独立完成，必须由人来主导的，它们共同构成了有效AI交互的五大支柱。

第一个支柱是“问题导向”。只有人能基于真实的业务需求、研究目标或个人困惑，来定义我们究竟要用AI解决“什么问题”，以及“为何要解决这个问题”。这是所有交互的根本起点。如果我们自己对问题都含糊不清，期望AI能给出精准的答案，那无异于缘木求鱼。AI可以帮助我们探索问题的不同方面，但最初那个具有价值的、值得投入资源去探究的问题，一定来源于人的思考和判断。

第二个支柱是“价值框架”。这里的价值框架，指的是我们选择以何种价值观、理论视角或分析框架来指导AI的思考与表达。比如，在分析一个社会现象时，是选择马克思主义的阶级分析视角，还是自由主义的个体选择视角？AI或许可以帮助我们梳理不同意识形态框架下的主要观点和表述方式，但最终做出选择，并承担这个选择所带来的立场与倾向，这完全是人的价值决断。不同的意识形态，会引导AI关注不同的信息，得出不同的结论，其重要性不言而喻。

第三个支柱是“信息储备”。常言道“巧妇难为无米之炊”，AI同样如此。有效的AI应用，高度依赖于用户是否有意识地积累、筛选、组织并提供高质量的“信息储备”。这些信息储备可能包括特定领域的背景知识、相关的案例数据、项目的历史文档，甚至是个人化的经验总结。如果我们自己没有积累与当前问题相关的信息库，或者提供给AI的信息杂乱无章、充满谬误，那么即便AI能力再强，也无法凭空创造出有价值的洞察。构建和管理好这个“信息储备库”，是人的一项重要工作。

第四个支柱是“理论方法”。选择和构建解决问题的理论及方法论，例如决定采用何种分析框架、研究路径或创作范式，是人类智慧的核心体现。比如，在进行一项复杂的调查研究时，是采用定量分析还是定性访谈，是先进行文献综述还是直接进行田野调查，这些方法论层面的决策，需要人的专业判断和经验积累。AI可以在我们确定的理论方法框架下，辅助执行具体的步骤，比如处理数据、归纳观点，但方法的原创性设计、理论的深刻理解与灵活应用，根本上依赖于人的主导。

第五个支柱是“表现形式”。AI可以生成多种多样的内容表现形式，比如报告、邮件、代码、诗歌、对话等等。但最终判断并选择何种结构、风格或格式最能有效地传递信息、达成沟通目的，并符合特定的受众和场景需求，这依然需要人的审美判断和策略考量。比如，同样的内容，是写成一篇严肃的分析报告，还是一篇生动的科普文章，亦或是一段简明扼要的演示文稿，这个选择权在人手中。

掌握这种以“知识工程”为核心的AI交互方法，其价值远不止于提升使用AI的效率和效果。更深远的意义在于，它能够反过来锻炼和提升我们自身的结构化思考能力、问题分解能力、信息辨别能力以及价值判断能力。当我们有意识地去构建这五个支柱时，实际上也是在梳理自己的思路，深化对问题的理解。

## 第 3 章：POMASA 与 MAS 的强状态

接下来讲 POMASA 与 MAS。这里同样不讲工具教程，而讲设计思想。很多人听到多智能体系统，会想到几个 Agent 互相聊天：一个负责想方案，一个负责写代码，一个负责审查，最后大家投票。这种理解太轻了。如果 MAS 只是多个 Agent 聊天，它很容易变成聊天接力。上一个 Agent 说了一堆，下一 个 Agent 摘要一下，再下一个 Agent 继续猜。状态在哪里，产物在哪里，谁负责验收，失败怎么回流，都不清楚。

真正有工程价值的 MAS，是强状态编排。它不是先问“有几个 Agent”，而是先问：任务的阶段是什么，每个阶段的输入是什么，输出是什么，产物放在哪里，谁有权限改，谁负责验收，失败后回到哪个阶段，下一步如何接手。Agent 只是角色，状态才是系统的骨架。

POMASA 的价值在这里就很好理解。它强调 Blueprint、Pattern、Prompt-Defined Agent、Orchestrated Pipeline、Filesystem Data Bus、Verifiable Data Lineage。把这些词翻译成工程语言，就是：角色要有协议，流程要有阶段，产物要有位置，文件系统要承担数据总线，数据来源和流转要能追踪，完成标准要能验证。

这和我们前面讲的软件基本功是一条线。Blueprint 像接口，Pattern 像可复用设计模式，Filesystem Data Bus 像团队共享的状态空间，Verifiable Data Lineage 像可审计的数据谱系。MAS 不是把复杂度交给更多模型，而是把复杂任务拆成可管理的角色、产物和状态转移。

可以用一个简单对比来理解。聊天接力是这样的：A Agent 输出一段总结，B Agent 读总结继续做，C Agent 再读 B 的总结做验收。每一步都依赖自然语言上下文，信息不断压缩，判断不断丢失。强状态流水线是另一种方式：资料收集产出 source inventory 和 raw notes；综合分析读取这些产物，写出 synthesis；引用核验读取 synthesis 和 source inventory，写出 verification；写作智能体读取 synthesis 和 verification，生成最终稿；每一步都有文件，有完成标准，有验收点。

这不是形式主义。文件系统作为数据总线，有一个重要好处：状态可恢复。对话断了，可以从文件恢复；换 Agent 了，可以从产物接手；要审计来源，可以沿着文件和引用找回去；发现错误，可以定位是资料收集错、综合分析错、核验错，还是写作阶段错。如果一切都在聊天里，错误就会变成一团模糊的上下文。

强状态也不是把所有东西都写下来。强状态是让关键产物有位置，关键转移有记录，关键判断有来源，关键验收有证据。写太多无效状态也会制造噪声。我们要记录的是系统继续推进所必需的信息：当前阶段、已完成产物、未解决问题、验收结果、失败原因、下游依赖。

MindFlow 和 POMASA 放在一起，就能看清 AI-native 工作系统的两面。MindFlow 更偏认知结构和自我演进，回答知识如何分层、能力如何沉淀、状态如何恢复。POMASA / MAS 更偏多角色、多产物、多阶段组织，回答谁做什么、产物如何流转、流程如何验收。两者共同指向一个结论：模型负责推理，流程负责约束，状态负责让推理结果可以被恢复、检查和交接。

## 第 4 章：状态控制三分法

在讲 Skill、工程文件和 Rules 之前，我们先建立一个控制框架。AI 工作系统里，控制方式大体可以分成三类：强约束固定、有建议的 AI 自主判断、完全由 AI 控制。

第一类是强约束、固定调度。比如 Rules、Hooks、必须读取的工程文件、固定检查点、必须执行的测试、提交前检查、权限边界和安全规则。这类控制可靠性最高，灵活性最低。它适合承载那些不能靠模型临场发挥的动作。比如测试失败不能宣称完成，引用材料必须可追溯，危险命令必须确认，提交信息必须符合规范。这些事情如果交给模型“记得做”，迟早会漏。

第二类是有建议的 AI 自主判断。Skill 就经常处在这一层。它不是每次都强制执行，而是根据任务触发。比如遇到调试任务，就应该加载系统化调试流程；遇到前端设计，就应该加载设计规范；遇到提交，就应该加载提交助手。这类方式保留了灵活性，也提供了能力，但它有一个天然问题：触发不稳定。模型可能漏触发，也可能误触发，或者触发得太晚。

第三类是完全由 AI 控制。模型自己决定读什么、怎么拆、什么时候停、是否验证。这类方式最灵活，适合探索、低风险实验和一次性任务。比如让模型帮你快速比较几个思路，或者临时整理一段材料，完全可以让它自由发挥。但如果任务进入多人协作、高风险交付、长期维护，这种自由就会变成不稳定。

这三类方式之间有张力。越固定，越可靠，但越不自由；越自由，越灵活，但越不稳定。工程设计的关键不是选一种，而是组合使用。强约束兜底，保证必须发生的动作；建议式调度提供可复用能力；AI 自主控制负责开放探索。

一个成熟的 AI 工作系统，应该能回答：哪些事情必须每次发生，哪些事情由任务触发，哪些事情可以交给模型探索。比如安全、权限、提交前检查、验收声明，应该偏强约束；写作风格、调研方法、调试流程，可以用 Skill 或工程文件提供建议式能力；头脑风暴、方案比较、探索性阅读，可以给模型更高自由度。

这里要避免一个误区：不要因为模型很强，就把所有控制都交给模型。模型越强，越容易让人产生“它会自己处理”的错觉。但工程系统要保证的是稳定性、可复查性和可交接性。稳定性往往来自确定性控制，而不是来自模型的好心情。

## 第 5 章：Skill、工程文件与 Rules

有了三分法，我们再看 Skill、工程文件与 Rules。它们本质上都是可执行知识。人的经验、判断标准、流程约束、输出格式，被组织成 Agent 可以读取、执行、检查的上下文资产。

Skill 往往像一个可复用能力包。它告诉 Agent：在某类任务里，应该按什么流程做，应该避免什么错误，应该产出什么结果。比如调试 Skill 会要求先复现问题、再定位根因、再做最小修复、最后验证；提交 Skill 会要求看 diff、看 status、生成规范提交信息、不要跳过 hooks。

但 Skill 的问题是调度不够确定。它通常依赖触发条件和模型判断。模型判断对了，它就加载；判断错了，它就漏掉。很多时候 Skill 内容本身没问题，问题在于没有被正确触发。所以 Skill 适合承载可复用能力，但不适合承载必须发生的安全动作。

工程文件更像流程里的固定上下文。比如 `CLAUDE.md`、`AGENTS.md`、需求模板、设计模板、任务状态文件、验收清单、项目原则文件。它们不一定叫 Skill，但在系统里承担类似作用：把团队约定、项目知识、执行入口和验收要求放在稳定位置。工程文件的优势是调度更确定，因为流程可以规定什么时候读取、读取哪份、产出到哪里。

Rules / Hooks 则更强。它们接近每次触发条件满足就执行的确定性控制。比如每次提交都跑 commitlint，每次完成前都跑验证，每次危险操作都要求确认。Rules 的好处是稳定，缺点是自由度低。如果规则太宽，会频繁打断流程；如果太窄，会漏掉风险；如果没有退出条件，会让系统僵硬。

所以三者要组合使用。Skill 提供能力，工程文件提供流程位置，Rules 提供兜底约束。把所有东西都写成 Skill，触发会不稳定；把所有东西都写成 Rules，系统会太僵硬；只靠工程文件，没有触发机制和检查机制，也会变成没人读的文档。

所以这一章更准确的主题，不是“成熟 Skill 像函数”，而是“Skill 构建及评定”。构建 Skill 时，不能只问这段提示词写得好不好，而要问它能不能进入评估体系。一个 Skill 至少要说清楚：适用的任务是什么，输入是什么，Agent 可以使用哪些工具，应该留下什么执行记录，最终环境应该变成什么状态，哪些结果算通过，哪些行为算失败。

这里可以借用 Agent eval 的语言来理解。Task 是一个独立任务，有输入和成功标准；Trial 是 Agent 对这个任务的一次尝试；Transcript 是完整过程，包括推理、工具调用、中间结果；Outcome 是最终状态；Grader 是评分逻辑；Evaluation Suite 是一组覆盖某类能力的任务集合。把这套语言放到 Skill 上，就会发现 Skill 不是“建议 Agent 怎么做”的文本，而是可以被任务、轨迹、结果和评分器检验的能力资产。

第一步是把 Skill 变成任务集合。很多 Skill 看起来写得很漂亮，但它只描述了理想过程，没有对应的任务样本。比如一个“代码审查 Skill”，不能只写“请检查可维护性、健壮性、安全性”。它应该有一组真实任务：一次小改动、一次大重构、一次安全敏感变更、一次测试缺失变更、一次文档型变更。每个任务都要有明确输入、期望产物和成功标准。这样我们才知道这个 Skill 到底覆盖什么能力。

第二步是记录 Trial 和 Transcript。Agent 执行 Skill 的过程不能只看最后输出。它有没有读取必要文件？有没有检查 diff？有没有运行验证？有没有在危险动作前停下来？有没有把失败写回任务状态？这些都需要通过 transcript 看。Anthropic 的 Agent 评估文章里特别强调要读转录轨迹，因为只看分数很容易误判：有时 Agent 结果错了，有时是评分器错了，有时是任务本身写得含糊。

第三步是看 Outcome。Agent 说“我完成了”不算完成，最终环境真的进入目标状态才算完成。对于代码 Skill，Outcome 可能是测试通过、diff 合理、没有引入本机路径、没有跳过 hooks；对于研究 Skill，Outcome 可能是关键事实都有来源、弱证据被降级、引用链可追溯；对于写作 Skill，Outcome 可能是结构完整、口语化、没有报告腔、章节主线一致。Outcome 把“模型自述”变成“环境状态”。

第四步是设计 Grader。评分器不能只有一种。能用代码判断的，就用确定性测试、静态检查、格式检查、状态检查；需要开放判断的，再用 LLM 评分器；涉及价值判断、业务判断和关键交付的，要保留人工抽样校准。这里有一个原则：尽可能用确定性评分器兜住客观部分，把模型评分器留给真正需要语义判断的部分。否则评估本身也会变成另一个不稳定 Agent。

第五步是形成 Evaluation Suite。一个 Skill 不能只在一个样例里表现好。它需要一组覆盖常见情况、边界情况、反例情况和回归情况的任务。能力评估问的是“这个 Skill 还能提升到哪里”，回归评估问的是“它以前能做对的事现在还做不做得对”。随着真实失败增加，失败要变成新的评估任务。这样 Skill 的改进才不是凭感觉，而是有任务集、有结果、有趋势。

所以，Skill 的评定可以分成几层。第一层看触发：该用的时候是否会用，不该用的时候是否少误触发。第二层看过程：执行轨迹里有没有读必要文件、调用必要工具、产生必要中间产物。第三层看结果：最终状态是否真的满足任务，而不是 Agent 自称完成。第四层看回归：同类任务反复运行时，是否稳定降低错误，而不是只在一个样例里表现好。

这也是为什么我说 Skill、工程文件和 Rules 都是知识对象。它们不是写给人看的说明书而已，而是要进入执行系统，影响 Agent 的行为。既然要影响执行，就必须有任务、有轨迹、有最终状态、有评分器，也要有失败回流。一个 Skill 如果在失败后不能产生新的测试、新的验收项或新的评估样例，它就没有完成知识工程闭环。

## 第 6 章：Knowledge 与 Story

接下来进入知识工程的核心。我想用两个词：Knowledge 和 Story。

在软件团队里，Story 通常被理解成一个需求条目，一个用户故事，一个待开发的功能增量。但在知识工程视角下，Story 更像一次知识变化。它不是静态文档，而是一段正在流动的知识：用户为什么要这个变化，当前系统哪里不满足，新的业务路径是什么，验收标准是什么，哪些规则要改变，哪些测试要增加。

Knowledge 则是这些变化被验证、被固化、被回写之后形成的当前状态。一个 Story 做完以后，如果只有代码变了，知识没有回写，团队其实没有真正吸收这次变化。下次另一个人或另一个 Agent 进来，还是不知道为什么这样设计，不知道边界在哪里，不知道哪些验收必须保留。

所以，本次分享采用一个核心观点：Story 是动态知识，Knowledge 是静态 Story。Story 是变化过程，Knowledge 是变化被吸收后的系统状态。

从这个角度看，user journey 本质上是团队发现、验证、固化知识的过程。需求澄清是在发现知识：用户真正想完成什么，哪些情况必须支持，哪些情况明确不支持。设计是在组织知识：哪些对象负责，边界怎么切，接口怎么定。实现是在编码知识：把判断和规则固化进系统。测试是在验证知识：系统是否真的符合预期。Review 是在校正知识：有没有遗漏风险、重复逻辑、错误抽象。验收是在确认知识进入当前状态。

如果一个 Story 做完了，但知识只留在聊天记录里，或者只留在某个 Agent 的临时上下文里，它就没有真正沉淀。高质量流程要求被验证的增量回写到 AC、ADR、工程文档、Skill、Capability、测试、eval case 或 workflow pattern 中。不是每个 Story 都要更新所有资产，但每个 Story 都应该问：这次变化有没有产生值得沉淀的新知识？有没有新的失败模式？有没有新的业务判断？有没有新的验收方式？

这也是 AI 编程和传统编程最大的差异之一。传统团队里，很多知识靠人记住；AI 工作系统里，不能指望 Agent 记住上一次对话。知识必须进入仓库，进入文件，进入测试，进入规则，进入 eval。否则每次任务都是重新解释，每次失败都是重新踩坑。

这里可以在现场用 Flyway Migration 做一个口头类比。Story 像一次带有时间性的增量变化，Knowledge 像所有增量执行后的当前状态。类比本身不需要单独展开成章节，关键是让听众记住：Story 完成后，知识必须进入当前状态。

## 第 8 章：Skill 与 Story 的验收

现在我们把 Skill 和 Story 放到同一个质量框架下看。高质量流程的关键不是“定义了 Skill”或“写了 Story”，而是它们能不能被稳定验收。

Skill 的验收，可以像验收函数一样问。什么时候应该触发？输入是否明确？前置假设是否说清？步骤能否被不同 Agent 稳定复现？输出是否结构化？下游是否能消费？是否内置验证动作？权限和副作用是否被限制？失败时如何处理？是否能减少重复解释？是否能防止同类错误复发？

如果这些问题回答不清楚，这个 Skill 就像一个形状不固定的盒子。看起来装了经验，实际无法稳定复用。今天某个 Agent 用得好，是因为上下文刚好；明天换一个任务，它可能误触发、漏步骤、产出下游不能用的结果。无法验收的 Skill，不是工程资产，只是经验片段。

Story 的验收也类似。一个 Story 不只是“实现某功能”。它要讲清楚业务意图、用户路径、验收标准、依赖关系、边界条件和最终状态。业务意图不清楚，Agent 可能做出功能但做错方向；用户路径不完整，测试可能只覆盖 happy path；验收标准不可测，最后只能靠感觉判断；依赖不明确，并行时就会互相踩；边界不清楚，模型就会自己补默认值。

这里要特别强调：不要只相信 Agent 说“我完成了”。Agent 的完成声明只是一个信号，不是证据。证据应该来自测试结果、浏览器验证、trace、outcome、代码 diff、引用核验、人工验收。反思循环也必须带证据。没有证据的反思，很容易变成模型写一段好听的总结。

失败也不应该只是失败。失败是 eval case 的来源。如果某次 Agent 漏掉了权限边界，这个失败应该变成新的测试、新的规则、新的 Skill 检查点，或者新的 Story 验收项。这样系统才会越跑越稳。否则失败只是一次返工，下次还会发生。

所以，Skill 与 Story 的验收，本质上都在回答同一个问题：这个知识对象能不能被不同人、不同 Agent、不同时间稳定消费？如果能，它就是工程资产；如果不能，它只是一次对话里的临时表达。

## 第 9 章：角色而非人

有了知识对象和验收，我们再看团队协作。这里的关键词是：角色而非人。

BA、DEV、QA、UI、Agent，不一定对应固定人头。它们更像几顶思考帽。一个人可以戴多顶帽子，一个 Agent 也可以承担某些帽子的动作。关键不是组织架构上有没有这些岗位，而是流程里有没有这些视角。

BA 关注业务意图和验收标准：用户到底要完成什么，业务规则是什么，哪些情况必须支持。DEV 关注设计和实现可行性：系统边界怎么切，接口怎么定，状态怎么维护，风险在哪里。QA 关注可测性和质量风险：验收标准能不能测，异常路径有没有覆盖，回归风险在哪里。UI 关注用户路径和交互完整性：用户从哪里进入，如何理解反馈，错误状态如何呈现。Agent 则承担检索、执行、修改、验证、总结等具体动作。

如果没有角色视角，AI 编程很容易变成“谁会写代码谁就往前冲”。模型开始实现，开发者开始修补，测试最后才发现业务路径不对，BA 再回来解释需求。这个流程看起来快，实际返工很多。角色视角的价值，是把不同问题提前暴露出来。

这里要引入单仓库原则：仓库是 BA / DEV / QA / UI / Agent 共享的强状态空间。仓库不只是代码存放处，而是需求、设计、计划、状态、测试、Review、规则、Skill、验收证据共同存在的地方。所有角色围绕同一个状态空间工作，才能减少信息损耗。

传统软件团队常见的问题，是部门墙、协调税和 PRD 信息损耗。业务知识通过会议和文档传给开发，开发再传给测试，测试再反馈给业务，中间每一跳都会丢信息。AI 的价值之一，是压低上下文传递成本和重复解释成本。Agent 可以读取同一套工程资产，沿着统一任务结构工作，通过自动测试和状态回写减少人工协调。

但这不等于“人越少越好”。重点不是替代团队，而是改变协作结构。人的角色更多转向目标、边界、反馈和治理；Agent 承担一部分检索、执行、验证和整理。角色不是被消灭，而是被重新组织。

## 第 10 章：统一流程与单线程协作

这是全场最重要的一章。前面所有概念，最终都要落到统一流程。

统一流程从需求进入开始。第一步不是编码，而是 Tasking。Tasking 是需求分析与拆解：把模糊需求变成可执行、可验证、可分派的任务结构。它要澄清业务意图、用户路径、验收标准、边界条件、依赖关系和风险。Tasking 与 SDD 很接近，都是把“先想清楚”制度化。

Tasking 之后是澄清。澄清的目标不是把所有问题问完，而是把影响实现和验收的未知项暴露出来。哪些信息必须问用户，哪些可以暂时假设，哪些必须写入风险，哪些可以延后验证，都要进入状态。好的澄清不是问很多问题，而是问对会改变实现方向的问题。

然后进入设计。设计不是大文档，也不是仪式。设计是把关键决策说清楚：对象边界是什么，接口契约是什么，状态如何变化，数据从哪里来，到哪里去，错误如何处理，测试怎么设计。对 Agent 来说，设计尤其重要，因为它决定模型的搜索空间。如果设计不清，模型会自己补设计，而且补得很快。

计划也不是 TODO 列表。计划是可执行路径：先做什么，后做什么，每一步的输入输出是什么，完成条件是什么，失败如何回流。一个好的 Plan 应该让不同 Agent 都能接手。它不是写给某个模型的临时提示，而是写给整个工作系统的执行协议。

实现阶段，Agent 可以发挥速度优势，但必须沿着设计和计划推进。实现过程中如果发现需求误差、设计缺口、测试缺口或工程约束冲突，不能只在对话里说一下，要更新状态，必要时回到澄清或设计。否则，系统表面继续前进，实际已经偏离轨道。

实现后进入 Review、测试和验收。Review 看工程风险，QA 看可测性和业务路径，UI 看用户体验路径，BA 看业务意图是否实现。测试不仅包括单元测试、集成测试，也可以包括浏览器模拟验证、关键路径回放、静态检查和 lint/type check。最后，代码出去，并且验收测试通过。

这条流程不是 checklist，而是资产生产线。每一步都有输入、输出、状态、负责人角色、检查点和回写位置。需求不是只流向代码，也流向 Knowledge。流程中发现的新规则、新失败模式、新业务判断、新测试场景，要同步更新 Skill、工程文件、Knowledge，或类似长期原则文件。

为什么先讲单线程？因为单线程闭环质量，是多线程和自动化的前提。如果一个需求从输入到输出都跑不稳，就不要急着并行；如果验收标准不清楚，就不要急着无人值守；如果状态不能恢复，就不要让 Agent 长时间跑。先让一个需求稳定地从输入到输出，能被验收，能回写资产。这个闭环跑通以后，才有资格谈规模化。

这里可以把统一流程想象成一条生产线，但它生产的不是代码，而是经过验证的工程资产。需求进入时通常是模糊的，有业务语言，有上下文，有隐含假设。Tasking 把它变成可执行任务；澄清把关键未知项暴露出来；设计把对象、接口和状态变化固定下来；计划把执行路径拆开；实现把知识编码；Review 和测试把错误挡住；验收把业务意图确认回来；最后回写把这次变化沉淀到 Knowledge。代码只是其中一个产物，不是唯一产物。

如果团队只盯着代码输出，就会忽略流程里最有价值的部分：那些被澄清出来的边界、被测试捕获的失败、被 Review 指出的风险、被验收确认的业务判断。这些东西如果不回写，下次又要重新解释。一个高质量流程的标志，是每做完一个需求，系统不只是多了一段代码，还多了一点可复用知识。

这也是为什么我反复强调“需求进来，代码出去，最后验收测试通过”。这句话看起来朴素，但它把 AI 工作系统拉回了软件工程本质。输入不能只是一句话，输出不能只是 diff，中间不能没有状态，结束不能只靠 Agent 自述。真正完成，是业务意图被实现，工程风险被检查，测试证据存在，知识资产被更新。

## 第 11 章：多线程协作流程

当单线程闭环稳定后，才谈多线程。

多线程流程通常从 Epic 到 User Story。Epic 是一个较大的业务目标，不能直接丢给 Agent 并行做。它要先被拆成多个 Story，每个 Story 有独立业务意图、验收标准和边界。拆完以后，还要识别依赖关系：哪些 Story 可以独立做，哪些必须等待基础能力，哪些共享同一模块，哪些会争用同一接口或数据结构。

这里要克制地讲“自动解析依赖”。这可以作为设计方向，但不能当成完全成熟的魔法。依赖识别需要工程资产支撑：模块边界清楚，接口契约明确，测试能覆盖关键路径，状态文件能记录每条线进展。没有这些基础，所谓自动并行只会制造冲突。

Worktree 是并行推进的重要隔离机制。每个 worktree 可以承载一个独立功能增量，让不同 Agent 在隔离环境中工作。但 worktree 不是随意并发。每条并行线仍然要有自己的 Story、Plan、State、测试、Review、合并门禁和知识回写。

多线程不是跳过统一流程，而是把统一流程复制到多个隔离工作流中，再通过依赖、状态和验收汇合。并行带来的提速，来自减少等待和协调损耗；质量仍然来自强状态、测试、Review、验收和回写。

这里要特别注意合并点。并行开发最容易在合并时暴露问题：两个 Story 改了同一个接口，一个 Story 改了数据结构，另一个 Story 还按旧结构写；一个 Agent 加了测试，另一个 Agent 改了业务路径但没有更新测试；一个 worktree 里验收通过，合并后系统整体失败。所以多线程流程必须有合并门禁。门禁不是只看冲突有没有解决，还要看依赖是否满足，测试是否重新跑，文档是否更新，状态是否汇合，知识回写是否完成。

多线程的另一个风险是局部最优。每个 Agent 在自己的 worktree 里都觉得完成了，但整体产品路径不通。这就需要更高层的 orchestrator 或者人工 gate 看全局：Epic 的业务目标是否被满足，Story 之间的接口是否一致，用户路径是否完整，最终验收是否覆盖整个增量。并行不是把整体责任拆没了，而是把局部执行并行化，再把整体验收拉回来。

这里最容易犯的错误，是把多线程理解成“让更多 Agent 同时写代码”。真正的多线程是“让多个可验收的功能增量在隔离状态下推进”。如果没有验收，越并行越乱；如果没有状态，越并行越难合并；如果没有依赖识别，越并行越容易互相覆盖。

所以，从单线程到多线程，是流程能力的扩展，不是速度按钮。只有当单线程流程稳定，知识对象清楚，验收标准可测，状态能恢复，才值得把它复制到多个 worktree 中并行推进。

## 第 12 章：自动化与无人值守

角色流程、强状态、工程文件和 Knowledge 都讲完后，我们才能讲自动化与无人值守。

这里要非常克制。无人值守不是完全自主魔法，而是长时间、低人工干预、有边界、有测试、有人工 gate 的开发模式。系统依靠明确任务边界、强状态、Worktree 隔离、自动测试、浏览器验证、可观测、人工 gate 和失败回流来运行。

长时间运行最怕的不是模型不会写代码，而是状态断掉。跨上下文窗口、跨 session、跨执行阶段时，Agent 必须知道现在做到哪里，哪些已经验证，哪些还没验证，下一步是什么，失败在哪里，能不能安全继续。如果这些信息只在聊天里，长时间运行很快会失控。

所以长时间任务需要 feature list、progress file、git history、clean state、单 feature 增量推进、自动测试和浏览器验证。Feature list 告诉系统要完成哪些增量；progress file 告诉系统当前做到哪里；git history 提供可审计的变化记录；clean state 让系统知道从哪里安全恢复；单 feature 增量避免一次铺太大；测试和浏览器验证提供完成证据。

低人工干预不等于没有人。人的 gate 仍然重要。高风险决策、需求歧义、权限变更、生产操作、验收判断，都应该有人工介入。自动化的目标不是把人拿掉，而是把人从重复解释、机械检查和上下文搬运中解放出来，让人聚焦目标、边界和反馈。

长时间运行也需要可观测。只看最终输出不够。我们要能看到 Agent 做了哪些步骤，调用了哪些工具，遇到哪些错误，如何修复，哪些测试通过，哪些测试失败，为什么判断完成。没有可观测，出了问题只能猜；有了可观测，失败可以进入 eval case，流程可以继续改进。

自动化的演进路径可以是：先有单线程统一流程，再有多线程 worktree 隔离，再有长时间低人工介入，再逐步探索 Agent Teams。Agent Teams 可以作为展望，但不应说成成熟的完全自治组织。

如果把无人值守讲得太神奇，团队很容易误解成“把需求丢进去，第二天代码自己出来”。这不是我想表达的。更现实的模式是：人定义目标和边界，系统把任务拆成可执行增量，Agent 在隔离环境里推进，测试和工具持续给反馈，遇到高风险决策或不确定需求时停下来请求人工确认。人不需要盯着每一行代码，但仍然控制方向、风险和验收。

长时间运行的核心不是让模型一直跑，而是让它每一段运行都可恢复、可检查、可停止。一个好的长时间任务，应该随时能回答几个问题：现在完成了哪些 feature？当前卡在哪里？最近一次验证是什么结果？工作区是否干净？如果现在中断，下一轮从哪里继续？如果这些问题回答不上来，就还不具备低人工干预的基础。

## 第 13 章：Agent 设计、提速、测试评估、AI 编排、工具层改造与展望

最后一章把具体实践放回统一流程。

先说 Agent 设计。Agent 不是万能助手。一个好的 Agent 应该从角色协议出发定义：它的上下文是什么，权限是什么，输入是什么，输出是什么，契约是什么，验收是什么，失败如何处理，副作用有哪些。不要一上来就造一个“全能工程师 Agent”。全能听起来强，实际边界最模糊，最难验收。

Agent 的权限尤其重要。Agent 是不受信任的操作员。不是说它有恶意，而是说它会犯错，会误解上下文，会过早收束，会把看起来合理的动作当成正确动作。所以工具层要最小权限，危险动作要确认，副作用要可控，输出要结构化，错误要可诊断。

再说提速。AI 的提速不只是 token 生成速度，也不是让模型多写代码。真正有组织价值的提速，来自上下文传递损耗下降、等待减少、反馈更快、测试更靠前、失败更快变成资产。传统团队扩大规模后会遇到沟通成本、同步成本、协调税和部门墙。AI 的价值之一，是让 Agent 共享仓库状态、读取同一套工程资产、沿着统一任务结构工作，并通过自动化验证和状态回写减少重复解释。

测试评估也要进入主线。Agent eval 不是只问“模型答得好不好”，而是设计 task、trial、grader、trace/transcript、outcome 和 evaluation suite。任务要明确，评估逻辑要可重复，过程轨迹要能回看，结果要能判断。失败是 eval case 的来源。模型“偷懒”很多时候不是态度问题，而是反馈不足、状态不足、验收不足。

可观测也是高质量流程的一部分。对 Agent 系统来说，只看最终输出不够，还要能看到中间轨迹、工具调用、错误回流和质量变化。否则你不知道它是正确推理后得到结果，还是误打误撞碰到了通过状态。可观测让系统可以诊断，也让团队知道下一步该改模型、改提示词、改工具、改测试，还是改流程。

AI 编排要从简单开始。路径固定、验收明确的任务，优先 workflow；任务开放、路径不确定、需要探索时，再增加 agent 自主性。Prompt chaining、routing、parallelization、orchestrator-workers、evaluator-optimizer、autonomous agents，都是工具箱里的模式，不是越复杂越先进。复杂度必须由任务需要驱动。

工具层也需要改造。人类体验和智能体体验不同。人喜欢页面、说明、宽松输入、可读输出；Agent 更需要 JSON、schema、字段裁剪、稳定错误码、权限边界、可自省接口和输入防御。一个对人友好的工具，不一定对 Agent 友好。未来很多工程系统会同时设计 human experience 和 agent experience。

举个例子，人看一个网页表格，可以通过视觉理解列含义、异常颜色、按钮位置；Agent 读同一个页面，可能需要结构化 DOM、稳定 selector、明确字段名、可机器读取的错误信息。人看到“保存失败，请稍后重试”还能猜测下一步；Agent 需要知道是权限问题、校验问题、网络问题，还是服务端错误。人可以容忍模糊提示，Agent 需要稳定协议。

所以，为 Agent 设计工具层，不是把现有人类界面暴露给它就完了。我们要给它可枚举的能力、可限制的权限、可审计的调用记录、可恢复的错误状态。确定性的事交给确定性的工具，工具输出尽量结构化，模型负责在结构化信息上做判断和推理。这样系统才不会把所有不确定性都堆到模型身上。

最后是展望。框架是为了消灭自己而存在，这是一个工程判断：当某个模式稳定后，它应该沉淀为工具、模板、规则、测试、Skill 或 workflow，减少人和 Agent 的重复解释。架构形态也会被 AI 重新塑造。AI 不是降低设计重要性，而是提高设计杠杆。未来我们会看到更多面向 Agent 的工程资产、更多可验证流程、更多低人工介入的长时间任务，也会看到 Agent Teams 这类组织形态探索。但无论形态怎么变，核心仍然是统一流程和高质量流程。

所以这一章不是一个工具清单，而是对前面主线的落地。Agent 设计解决角色和权限，提速解决协作成本，测试评估解决完成证据，编排解决任务结构，工具层改造解决 Agent 如何稳定操作世界。它们看起来分散，其实都服务于同一个目标：让 AI 不只是更快地产生输出，而是在工程系统里持续地产生可信输出。

## 结尾

今天我们从工程化与 SDD 讲起，回到 SoC、抽象和目录结构，再看 MindFlow 的知识分层、POMASA 的强状态 MAS，接着讲状态控制三分法、Skill / 工程文件 / Rules、Knowledge 与 Story、Flyway 类比、验收、角色流程、多线程、自动化和工具层改造。

如果只带走一句话，我希望是这一句：AI-native workflow 是模型推理、流程约束和状态反馈的动态系统。模型能力会继续增强，但工程系统仍然需要边界、状态、验收和回写。我们要做的，不是让 AI 更快地产生更多代码，而是让 AI 在一个高质量流程里持续产出可验证、可交接、可演进的工程资产。

这也是我希望大家回到团队后真正开始做的事：不要先追逐更多工具，而是先检查自己的流程有没有入口、状态、验收和回写。只要这四件事稳住，模型能力越强，团队收益越大；如果这四件事缺失，模型越快，混乱也会越快。这就是知识工程的现实价值，也是这场分享最终的落点。谢谢。

## 附录 / Q&A 备选

### Q1：Prompt Engineering 还重要吗？

重要，但它越来越像任务接口设计。提示词不再只是“怎么问模型”，而是定义输入、输出、约束、角色、上下文和验收。对于高风险、长期运行、多人协作的任务，仅靠一次提示词不够，需要工程文件、Rules、测试和状态回写。

### Q2：什么时候用 workflow，什么时候用 agent？

如果路径固定、验收明确、风险较高，优先用 workflow。如果任务开放、路径不确定、需要探索，可以增加 agent 自主性。Anthropic 的建议是从简单模式开始，只有需求推动时才增加复杂度。

### Q3：如何开始建设统一流程？

先不要做大平台。选一个真实需求，跑通单线程闭环：Tasking、澄清、设计、计划、实现、Review、测试、验收、回写。把每一步的输入输出、状态文件、责任角色和检查点固定下来。等单线程稳定后，再谈 worktree 并行和长时间运行。

### Q4：Eval 工具怎么选？

Promptfoo、Braintrust、LangSmith、Langfuse 等工具可以进一步调研；SWE-bench、Terminal-Bench 等 benchmark 也可以作为背景材料。但本稿不做工具推荐或排名。更重要的是先定义自己的 task、grader、trace、outcome 和 evaluation suite。

### Q5：如何判断一个 Skill 是否值得沉淀？

看它是否反复出现，是否能减少重复解释，是否能防止同类错误，是否有稳定触发条件，是否有明确输入输出，是否能被不同 Agent 复现，是否能被下游消费，是否有内置验证。
